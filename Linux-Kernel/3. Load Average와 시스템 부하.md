
### 3.1 Load Average의 정의

`man proc`을 통해 확인한 Load Average의 정의는 다음과 같다.

- **정의:** _The first three fields in this file are load average figures giving the number of jobs in the run queue (state R) or waiting for disk I/O (state D) averaged over 1, 5, and 15 minutes._
    
- 1분, 5분, 15분 동안의 **R 상태(실행 중/대기)와 D 상태(I/O 대기) 프로세스 개수의 평균값**이다.
    
- **상대성:** 프로세스 수를 세는 것이기 때문에, 시스템의 **CPU 코어 수**가 몇 개냐에 따라 동일한 Load Average 값이라도 그 의미가 전혀 다르다.
    - **코어 1개일 때 Load 2:** 처리 가능한 프로세스보다 대기 중인 프로세스가 더 많은 과부하 상태
    - **코어 2개일 때 Load 2:** 시스템이 처리 가능한 만큼의 적절한 프로세스가 있는 상태
        
### 3.2 Load Average 계산 과정

- Load Average라고 하면 가장 먼저 떠오르는 명령어는 **`uptime`** 이다.
- ```sh
  $ uptime
	 15:22:14 up 2 min,  1 user,  load average: 0.01, 0.01, 0.00
  ```
    
- **추적 방법:** `strace` 명령어로 `uptime`을 분석해 보면, 직접 계산하는 것이 아니라 `/proc/loadavg` 파일을 열어서 내용을 읽는 것을 확인할 수 있다
	- ```sh
	  $ strace -s 65535 -f -t -o uptime_dump uptime
		 15:23:27 up 3 min,  1 user,  load average: 0.00, 0.00, 0.00
	  ```
	- ```sh
	  628  15:23:27 openat(AT_FDCWD, "/proc/loadavg", O_RDONLY) = 4
	  2628  15:23:27 lseek(4, 0, SEEK_SET)    = 0
	  2628  15:23:27 read(4, "0.00 0.00 0.00 1/114 2628\n", 8191) = 26
	  2628  15:23:27 fstat(1, {st_mode=S_IFCHR|0620, st_rdev=makedev(0x88, 0), ...}) = 0
	  2628  15:23:27 write(1, " 15:23:27 up 3 min,  1 user,  load average: 0.00, 0.00, 0.00\n", 61) = 61
	  ```
    - 덤프파일 하단부에 `/proc/loadavg`  파일을 열어서 이를 읽고 화면에 출력해주는 것을 확인 할 수 있다.
    - 즉, 직접 Load Average 값을 계산하는 것이 아니고, 커널이 미리 준비해 둔 `/proc/loadavg` 파일을 단순히 읽어서 보여주는 명령어이다.

- **상세 계산 메커니즘:**
	- ![](images/Pasted%20image%2020260210001238.png)
		- [그림 20. Load Average의 계산 과정, DevOps와 SE를 위한 리눅스 커널 이야기, p.47](https://www.yes24.com/Product/Goods/44376723)
    - **Tick(틱)의 개념:** 하드웨어 타이머가 일정 간격으로 보내는 신호이자 커널의 최소 시간 단위(심장박동)이다. 커널은 이 주기마다 깨어나 시스템 상태를 점검한다.

    - **실시간 누적 (`cpu_load_account_active`):** 
	    - 매 Tick 주기마다 현재 CPU의 `nr_running`(R)과 `nr_uninterruptible`(D) 프로세스 개수를 세어 `nr_active` 변수에 담는다.
	    - `nr_active` 값이 기존에 계산된 값과 다를 경우, 그 차이(Delta)값을 구한후 전역 변수(`calc_load_tasks`)에 입력한다.
        
    - **주기적 평균 산출 (`calc_global_load`):** 5초 간격으로 호출되어 누적된 데이터를 바탕으로 1분, 5분, 15분 단위의 지수 이동 평균을 계산하고 `avenrun[]` 배열에 저장
	    - **이원화 목적:** 빈번한 '기록(Tick)'과 복잡한 '통계 계산(5초)'을 분리하여 CPU 성능 저하를 방지하면서 정확한 부하 지표를 유지
        

### 3.3 CPU Bound vs I/O Bound

Load Average 수치만으로는 어떤 종류의 부하가 발생하는지 알 수 없다.

- **CPU Bound 부하:** 무한 루프 연산과 같이 CPU 자원을 많이 사용하여 발생하는 부하(`nr_running` 증가)이다.
    
- **I/O Bound 부하:** 디스크 읽기/쓰기 작업이 많아 I/O 응답을 대기하며 발생하는 부하(`nr_uninterruptible` 증가)이다.
    
- 부하의 종류에 따라 해결 방법이 다르므로 원인을 정확히 파악하는 것이 중요!
    

### 3.4 vmstat으로 부하의 정체 확인하기

Load Average가 '부하의 유무'를 알려준다면, `vmstat`은 '부하의 성격'을 상세히 알려주는 도구이다.

- **r 열:** 실행되기를 기다리거나 현재 실행 중인 프로세스의 개수(`nr_running`)
	- 이 값이 높으면 **CPU 자원 경합**이 심한 상태
    
- **b 열:** I/O를 위해 대기열에 있는(D 상태) 프로세스의 개수(`nr_uninterruptible`)
	- 이 값이 높으면 **디스크/네트워크 저장소 병목** 상태

- **si/so (swap in/out):** 메모리 부족으로 스왑 영역을 사용하는 정도 
	- 0이 아니면 메모리 증설이 시급

- **bi/bo (block in/out):** 초당 블록 장치(디스크)에서 읽고 쓰는 양
	- I/O 부하의 크기를 직접적으로 보여줌.

- **us/sy/id/wa (cpu):** 유저/시스템/유휴/I/O 대기별 CPU 점유율
	- `wa`가 높다면 CPU는 놀고 있지만 I/O 응답이 늦어 시스템이 느려진 상태
    
- **분석 방법:** Load Average가 높을 때 `r` 값이 높으면 CPU 부하, `b` 값이 높으면 I/O 부하로 판단 가능하다.
    

### 3.5 Load Average가 시스템에 끼치는 영향

- **CPU 기반 부하:** 연산 프로세스들이 CPU를 점유하여 실제 요청을 처리해야 할 프로세스들이 실행 지연을 겪으며 응답 속도가 현저히 느려진다.
    
- **I/O 기반 부하:** 프로세스들이 I/O 대기를 위해 **D 상태**에 빠져 있으므로 CPU 경합은 상대적으로 덜하며, CPU 부하 때보다는 응답 속도가 빠를 수 있다.
    
> **핵심 포인트:** 프로세스가 어떤 자원을 쓰느냐에 따라 시스템 성능에 미치는 영향은 다르다. 단순히 Load가 높다고 장비를 증설하는 것만이 해결책은 아니다.

### 3.6 Case Study - OS 버전과 Load Average

- **현상:** CentOS 7.2와 CentOS 6.5 간 동일 애플리케이션임에도 Load Average 수치 차이가 발생했다.
	- 동일한 자바 애플리케이션이기에 프로세스 개수가 다를 가능성은 적으며, 지표 수집 방식의 차이를 의심

- **검증 테스트:**
	- Load Average 수치가 다른데, 두 서버가 정말 동일한 작업을 하고 있는지 어떻게 확신할 수 있는가?
		- **`vmstat` 명령**을 활용
			- `vmstat`의 **r(runqueue)** 과 **b(blocked)** 열은 커널의 스케줄러 정보를 직접 참조하여 현재 동작 중인 프로세스/스레드 개수를 있는 그대로 출력한다.
			- **결과:** 테스트 시 두 서버의 Load Average는 달랐지만, **`vmstat`의 r 열 값은 두 서버 모두 동일**하게 측정됨
				- 이를 통해 서버 내부에서 돌아가는 스레드 개수는 같으나 'Load Average 지표'만 한쪽에서 누락되고 있음을 확인함
	- 스레드 기준 테스트
		- **단일 스레드 테스트:** 무한 루프 연산(CPU Bound) 코드로 테스트 시 두 서버 모두 Load가 약 1.0으로 정상 측정되었다.
		- **멀티 스레드 테스트:** 10개 스레드를 생성하는 연산 코드로 테스트 시 CentOS 7.2는 약 10.0으로 정확히 측정되었으나, CentOS 6.5는 수치가 거의 상승하지 않았다.
- **원인 분석 (`/proc/sched_debug`):**
	- `sched_debug` 파일 내 `nr_running` 값을 확인한 결과, 실제로는 두 서버 모두 10개의 스레드가 Run Queue에서 정상적으로 동작 중임을 확인.
	- 이는 CentOS 6.5(커널 2.6.32 일부 버전)의 **틱리스(Tickless) 커널 버그** 로 판명됨 
		- 시스템이 유휴 상태라고 착각하여 타이머 인터럽트(Tick)를 발생시키지 않았고, 이로 인해 5초마다 실행되는 Load Average 계산 함수가 누락된 스레드들을 카운트하지 못한 것
	- Tickless
		- 리눅스의 핵심 전력 관리 기술
		- 과거에는 CPU가 할 일이 없어도 매 Tick(심장박동)마다 깨어나서 상태를 확인했다면
		- 틱리스 커널은 할 일이 없으면 타이머 인터럽트를 중단시켜 CPU가 깊은 수면(C-state)에 들게 한다,
    
- **결론:** 커널 지표가 항상 완벽하지 않으므로 `vmstat`, `sched_debug` 등 다양한 툴을 조합하여 교차 검증해야 한다.
    

### 3.7 요약

1. **Load Average 정의:** 실행 중 혹은 실행 대기이거나 I/O 작업을 위해 대기 큐에 있는 프로세스들의 수이다.
    
2. **해석 방법:** 시스템의 CPU 코어 수를 기반으로 한 상대적인 값으로 해석해야 한다.
    
3. **지표 신뢰성:** 커널 버그 등으로 지표가 부정확할 수 있으므로 절대적으로 신뢰해서는 안 된다.
