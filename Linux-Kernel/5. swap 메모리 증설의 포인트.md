
### 5.1 swap 영역

- **정의**: 물리 메모리(RAM)가 부족할 경우를 대비하여 디스크의 일부를 메모리처럼 사용하도록 확보해 놓은 비상용 공간
    
- **필요성**: 프로세스가 연산을 위한 메모리 공간을 확보하지 못하면 시스템이 응답 불가 상태에 빠질 수 있으므로, 시스템의 안정적 운영을 위해 필수적이다.
    
- **특징**: 디스크를 사용하기 때문에 실제 물리 메모리에 비해 접근 및 처리 속도가 현저하게 떨어진다.
    

**`free -k` 명령으로 swap 확인**

```sh
[ec2-user@ip-172-31-4-90 ~]$ free -k
               total        used        free      shared  buff/cache   available
Mem:          469112      118520      105540         452      245052      337884
Swap:         468988           0      468988

```

1. **total**: 전체 swap 영역의 크기. 
    
2. **used**: 현재 사용 중인 swap의 크기. 아주 적은 양이라도 swap이 사용되기 시작했다면 시스템 메모리 상태를 점검해야 한다.
    
3. **free**: 현재 남아있는 swap 영역의 크기.
    
- **성능 저하의 신호**: swap 영역을 사용한다는 것 자체가 시스템 메모리 부족을 의미한다. 
	- 누가 swap을 사용하는가?
		- swap 발생 시 단순히 메모리 증설만 고려할 것이 아니라, **어떤 프로세스가 swap을 유발하는지** 확인하는 것이 필수적이다.
		- **관리 용도 프로세스의 누수**: 서비스 본연의 기능이 아닌 모니터링이나 로그 수집 등 관리용 프로세스에서 메모리 누수가 발생해 swap을 점유하는 경우가 있음.
		- **해결 방법**: 이런 경우라면 해당 관리 프로세스를 종료(Kill)하는 것만으로도 서비스 성능 저하를 즉각적으로 해결할 수 있음.
		- swap의 사용 여부만큼이나 **"누가 swap을 사용하는가"** 가 시스템 운영의 매우 중요한 판단 기준이 된다.

### 5.2 버디 시스템 (Buddy System)

![](images/Pasted%20image%2020260302210449.png)
그림 5-1 (출처.devops와 se를 위한 리눅스 커널이야기)

- **개념**: 커널이 물리 메모리를 할당할 때 사용하는 알고리즘으로, 연속된 메모리 영역을 관리한다.
    
- **동작 원리**: 
	- 커널은 물리 메모리를 연속된 페이지 덩어리로 관리하며, 이를 '버디'라고 부른다.
	- 즉, 커널은 메모리를 $2^n$ 단위(4KB, 8KB, 16KB 등)의 "버디"로 관리한다. 
	- **할당 방식**: 프로세스가 특정 크기의 메모리를 요청하면, 커널은 $2^n$ 단위(4KB, 8KB, 16KB 등) 중 가장 적합한 크기의 연속된 공간을 할당한다.
		- 예) 8KB 요청 시 4KB 두 개가 아닌, 연속된 8KB 영역 하나를 내어줌으로써 메모리 단편화를 방지하고 응답 속도를 높인다.
	- 장점: 메모리 단편화(Fragmentation)를 방지하고 할당 및 해제 속도를 극대화함
    
- **상태 확인**: `/proc/buddyinfo`를 통해 각 존(Zone)별 가용 메모리 상태를 확인 가능하다.

```sh
$ cat /proc/buddyinfo
Node 0, zone      DMA      1      0      0      1      2      1      1      1      0      1      3
Node 0, zone    DMA32   1296    838    470    182    133     70     35     14     14      6     42
Node 0, zone   Normal    238    391    186     76     39     10      3      1      3      1      0
```

- 이 파일은 현재 커널이 보유한 각 크기별(Order) 가용 메모리 블록의 개수를 보여준다.
- **컬럼의 의미**: 왼쪽부터 순서대로 $2^0, 2^1, 2^2 \dots 2^{10}$ 개의 페이지 덩어리를 의미한다.
- **계산법 (DMA 존 기준)** :
	- 첫 번째 열(1): $4\text{KB} \times 1$개 = $4\text{KB}$
	- 두 번째 열(0): $8\text{KB} \times 0$개 = $0\text{KB}$
	- 세 번째 열(0): $16\text{KB} \times 0$개 = $0\text{KB}$
	- 네 번째 열(1): $32\text{KB} \times 1$개 = $32\text{KB}$
	- ... (이런 식으로 모든 열을 합산하여 해당 Zone의 가용 메모리 총량을 구함)
- 메모리 단편화 진단 포인트
	- **오른쪽 열의 중요성**: 오른쪽으로 갈수록 큰 단위($2^9, 2^{10}$)의 연속된 메모리.
	- **단편화 발생 신호**: 전체 가용 메모리(`free`) 양은 충분한데 **오른쪽 열의 숫자들이 0에 가깝다면**, 큰 덩어리의 연속된 메모리가 없다는 뜻
	- **실무적 현상**: 
		- 이 경우 시스템은 새로운 프로세스를 띄우거나 큰 I/O 작업을 할 때 메모리를 재배치(Compaction)하느라 순간적인 성능 저하(Stall)가 발생할 수 있음.
		- 커널은 시스템 전체를 뒤져서 흩어져 있는 작은 조각들을 한곳으로 모아 큰 덩어리를 만드는 **'메모리 재배치(Compaction)'** 작업을 수행하기 때문
- **`buddyinfo`와 `free` 명령의 관계** 
	- `free` 명령이 "전체 잔량"이라는 총량을 보여준다면, `buddyinfo`는 그 잔량의 **"질(Quality)"** 을 보여준다.
	- 총량이 1GB나 남았어도 모두 4KB 단위로 쪼개져 있다면(왼쪽 열에만 숫자가 쏠려 있다면), 커널은 매우 불안정한 상태라고 판단해야 한다. 
		- 특히, 큰 단위 블록이 고갈되었다면, 이는 시스템을 오래 가동했을 때 발생하는 전형적인 현상이다.
- 메모리 존(Zone)의 구분 원리
	- **DMA** : 16MB 이하의 아주 낮은 주소 영역. 고전적인 장치들의 호환성을 위해 유지됨.
	- **DMA32** : 16MB ~ 4GB 사이의 주소 영역. 32비트 기반 장치들이 접근할 수 있는 영역
	- **Normal** : 4GB 이상의 주소 영역. 일반적인 프로세스의 데이터가 할당되는 주 영역
- 커널은 메모리 요청이 들어오면 가급적 높은 주소(Normal → DMA32 → DMA)부터 사용하려고 노력한다.
	- **하위 호환성 유지**: 낮은 주소 영역인 `DMA`나 `DMA32`는 구형 하드웨어나 특정 32비트 장치들이 접근할 수 있는 **유일한 통로**인 경우가 많다.
	- **영역 보존**: 만약 일반적인 프로세스들이 낮은 주소(DMA)부터 다 써버리면, 나중에 정작 그 영역이 꼭 필요한 장치 드라이버가 메모리를 할당받지 못하는 상황이 발생하기 때문
- **메모리 할당 테스트**: `malloc()`을 이용해 메모리를 요청하면 해당 크기에 맞는 버디 리스트에서 숫자가 줄어드는 것을 확인할 수 있었다.

### 5.3 메모리 재할당 과정

![400](images/Pasted%20image%2020260302212935.png)
그림 5-2 (출처.devops와 se를 위한 리눅스 커널이야기)

- **커널의 메모리 관리 전략**: 커널은 가용 메모리를 최대한 캐시(Page Cache, Buffer 등) 용도로 사용하여 성능을 높이려 한다.
    
- **재할당 로직**:
    
    1. **캐시 메모리 비우기**: 프로세스가 메모리를 요청할 때 가용 메모리가 부족하면, 가장 먼저 커널이 사용 중인 캐시 영역을 해제하여 반환한다.
        
    2. **Swap 사용**: 캐시를 비워도 메모리가 모자라면, 프로세스가 사용 중인 메모리 중 `Inactive List`에 있는 것을 swap 영역으로 이동시키고 해당 물리 공간을 재할당한다.
        
- **I/O 발생과 성능**: swap 재할당 과정은 디스크 I/O를 수반하므로 시스템 성능 저하의 주원인이 된다. 
- `dd` 명령을 이용한 메모리 변화 테스트
	- **1단계: 대용량 파일 생성 (`dd` 실행)**
		- **명령**: `dd if=/dev/zero of=./file_1 bs=1024 count=1000000` (약 1GB 파일 생성)
		- **현상**: 파일을 생성할 때마다 커널은 이 데이터를 **`cached`** 영역에 쌓는다.
		- **결과**: 파일을 여러 개(7개 등) 생성하면, `free` 명령 결과에서 `used`와 `cached`가 급격히 늘어나고 `free` 영역은 바닥을 드러내게 된다.
	- **2단계: 신규 메모리 할당과 캐시의 실시간 해제**
		- **상황**: `cached`가 가득 차고 `free`가 거의 없는 상태에서, 새로운 프로세스가 메모리 할당을 요청한다.
		- **`vmstat` 변화** :
			- **캐시 감소**: `cache` 수치가 약 6GB에서 5GB, 4GB로 실시간 감소함
			- **free 유지**: 가용 메모리(`free`) 수치는 거의 변하지 않는데, 이는 커널이 캐시를 비우자마자 신규 프로세스에게 즉시 할당하기 때문
			- **used 증가**: 캐시를 제외한 실제 메모리 사용량은 할당받은 만큼 정비례하여 늘어남
     - **3단계: 캐시 임계점 도달 및 Swap 발생**
		- **상황**: 페이지 캐시를 더 이상 줄일 수 없는 최저 임계점(약 220MB)에 도달함
		- **커널 동작**: 비울 캐시가 없으므로 기존 프로세스가 사용 중인 메모리를 **Swap 영역**으로 밀어내기 시작
		- **vmstat 핵심 변화** :
		    - **swpd 증가**: 사용 중인 swap 크기가 0에서 급격히 증가함
		    - **so (Swap Out)**: 메모리 내용이 디스크(Swap)로 나가는 작업이 초당 수십만 KB 단위로 발생함
		    - **`si` (Swap In) '0' 유지**: 현재는 메모리를 비워주는 단계이므로, 디스크에서 메모리로 데이터를 다시 가져올 필요가 없어 `si`는 발생하지 않음
	- **테스트 결과의 의미**
		- **커널의 우선순위**: 커널은 `free` 메모리가 부족해지면, 당장 프로세스를 죽이거나 swap을 쓰기 전에 **사용 중인 페이지 캐시를 먼저 비워서** 프로세스에게 넘겨준다.
		- **성능 분기점**: `cache`가 줄어드는 단계에서는 성능 저하가 크지 않으나, `so`가 발생하는 swap 단계부터는 디스크 I/O로 인해 시스템 응답 속도가 현저히 떨어진다.

### 5.4 vm.swappiness와 vm.vfs_cache_pressure

커널은 두 가지 주요 파라미터를 통해 메모리 재할당 동작을 조절한다.
#### 1) vm.swappiness

- **정의**: 커널이 얼마나 공격적으로 메모리 페이지를 swap 영역으로 옮길지 결정하는 값
	- ```
	  This control is used to define how aggressive the kernel will swap memory pages. Higher values will increase aggressiveness, lower values decrease the amount of swap. The default value is 60.
	  ```
- **동작 원리**:
    
    - **기본값은 60**이며, 0에서 100 사이의 값을 가진다.
        
    - **값이 클수록(예: 100)**: 페이지 캐시를 보존하려 하며, 대신 프로세스의 메모리 영역을 더 빨리 swap으로 이동시킴.
        
    - **값이 작을수록**: 가능한 한 페이지 캐시를 비우는 작업을 먼저 진행함
        
- **설정값을 제공해주는 이유**: 페이지 캐시는 디스크 I/O 접근을 줄여 응답 속도를 높이므로, 무조건적인 캐시 해제보다 자주 쓰이지 않는 메모리를 swap으로 내리는 것이 시스템 전체 성능에 유리할 수 있기 때문
    
#### 2) vm.vfs_cache_pressure

- **정의**: 디렉터리나 inode에 대한 캐시(dentry/inode cache)를 재할당하려는 경향을 조절한다.
	- ```
	  This percentage value controls the tendency of the kernel to reclaim the memory which is used for caching of directory and inode objects
	  ```
- **핵심 메커니즘 분석**:
	- 커널 소스(`fs/dcache.c`)의 `shrink_dcache_memory()` 함수 내 재할당 공식은 다음과 같다.
$$return\ (dentry\_stat.nr\_unused\ /\ 100)\ *\ sysctl\_vfs\_cache\_pressure;$$
- **공식의 의미**:
	- **`dentry_stat.nr_unused`**: 현재 사용되지 않는 dentry 객체의 개수이다.
	  
	- **기본값(100) 설정 시**: 사용하지 않는 객체 개수만큼 그대로(100%) 반환하여 재할당에 사용한다.
	  
	- **100보다 클 때(예: 10,000)**: 실제 미사용 객체보다 훨씬 더 많은 양을 재할당하도록 커널에 압력을 넣는다. 결과적으로 dentry 캐시가 매우 빠르게 삭제된다.

	- **100보다 작을 때**: 미사용 객체 중 일부만 반환하도록 하여 캐시를 더 오래 유지한다.
- **특징**:
    
    - **기본값은 100**이며, 0으로 설정하면 dentry/inode 캐시를 절대 반환하지 않아 메모리 부족 현상을 일으킬 수 있으므로 주의해야 한다.
        
    - **값이 클수록(예: 10000)**: dentry 캐시를 매우 공격적으로 반환함 
        
    - I/O가 빈번한 서버에서 이 값을 조정하여 캐시 반환 효율을 높일 수 있음.

### 5.5 메모리 증설의 포인트

- **증설 판단 기준**: swap 사용이 발생한다고 무조건 증설하는 것이 아니라, 원인을 먼저 파악해야 한다.
    
- **판단 유형 (그래프 분석)**:
    
    1. **선형적 증가 (그림 5-3)** : 메모리 사용량이 시간이 지남에 따라 계속 늘어난다면 **메모리 누수(Memory Leak)** 를 의심해야 한다. 이 경우 증설해도 결국 다시 swap을 사용하게 된다.
        
    2. **순간적 폭증 (그림 5-4)**: 특정 시점에만 요청이 몰려 폭증하는 경우도 있다. 
	    - 서비스 응답 속도에 치명적이지 않다면 swap으로 방어할 수 있으나, 안정적인 서비스를 위해 최대치에 맞춘 증설이 도움이 된다.

### 5.6 Case Study - gdb를 이용해서 메모리 누수 잡기

- **분석 도구 활용**: 메모리 누수가 의심될 때 `pmap` 명령으로 프로세스의 메모리 영역별 크기를 확인하고, `gdb`를 통해 힙(Heap) 영역의 메모리 덤프를 생성하여 로직 문제를 예측할 수 있다.
    
- **테스트 프로그램**: 1MB 영역을 `malloc()`으로 계속 할당하고 해제하지 않는 루프를 통해 누수 상황을 재현함.
	- **테스트 프로그램 구조**: 1MB 영역을 `malloc()`으로 할당받은 후 `sprintf()`를 통해 현재 시간을 기록하고, 할당된 메모리를 해제(`free`)하지 않은 채 무한 루프를 도는 방식.
	- ```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/time.h>

#define MEGABYTE 1024*1024

int main() {
    struct timeval tv;
    char *current_data;

    while (1) {
        gettimeofday(&tv, NULL);
        current_data = (char *) malloc(MEGABYTE);
        sprintf(current_data, "%ld", tv.tv_usec);
        printf("current_data = %s\n", current_data);
        sleep(1);
    }

    exit(0);
}
	  ```
    
- **결과 확인**: `ps aux` 명령을 통해 해당 프로세스의 **RSS(Resident Set Size)** 가 계속 증가하는 것을 확인할 수 있음.
	- `ps`는 현재 실행 중인 프로세스(Process)의 상태를 보여주는 명령어, `aux` 옵션을 붙여 시스템 전체의 상세 정보를 확인
	- `RSS`는 프로세스가 **실제 물리 메모리(RAM)에 점유하고 있는 크기**를 의미
	- RSS 12748 -> 12756
		- 프로그램이 특별한 작업을 더 하는 것이 아닌데도 **RSS 수치가 멈추지 않고 계속 커진다면**, 메모리를 할당만 하고 해제(free)하지 않는 **메모리 누수(Memory Leak)** 상태임을 판단할 수 있음.
	- ```sh

### 5.7 5장 요약

1. **버디 시스템**: 커널은 연속된 메모리 페이지 수를 기준으로 메모리를 할당하고 관리함.
    
2. **swap의 역할**: 메모리가 부족할 때 프로세스가 사용하는 메모리를 해제하여 저장하는 공간이며, 이 과정에서 발생하는 I/O로 인해 시스템 성능 저하가 일어남.
    
3. **swap 발생의 의미**: 시스템의 물리 메모리가 실제로 부족하거나, 특정 프로세스에서 **메모리 누수**가 발생하고 있다는 강력한 신호임.
    
4. **모니터링 도구**: swap을 어떤 프로세스가 사용하는지 정확히 알기 위해 `smem`과 같은 유틸리티를 활용하는 것이 효과적임.
    
5. **vm.swappiness**: 메모리 재할당 시 페이지 캐시를 먼저 비울지, 아니면 프로세스 메모리를 swap으로 보낼지의 비율을 조절하는 파라미터임.
    
6. **vm.vfs_cache_pressure**: 페이지 캐시 중에서도 디렉터리 캐시(dentry)나 inode 캐시를 얼마나 공격적으로 해제할지 결정함.
