
### 1. 메모리 사용량 확인하기

메모리는 프로세스가 연산할 때 필요한 함수, 변수 등을 저장하는 공간이다.
메모리가 부족하면 시스템 응답 불가나 큰 성능 저하가 발생하므로, CPU 사용률만큼이나 중요하게 관리되어야 한다.

```sh
[ec2-user@ip-172-31-4-90 ~]$ free -m
               total        used        free      shared  buff/cache   available
Mem:             458         113         112           0         231         332
Swap:            457           0         457

```

- **`free` 명령:** 리눅스에서 메모리 현황을 가장 빠르게 확인할 수 있는 도구
    - **주요 옵션:** `-b`(byte), `-k`(KB), `-m`(MB), `-g`(GB) 단위로 출력 가능.
        
- **Mem:**
    1. `total`: 시스템에 설치된 전체 메모리 양
    2. `used`: 현재 시스템에서 사용 중인 메모리 양
    3. `free`: 아직 사용되지 않은 순수한 여유 메모리
    4. `shared`: 프로세스 사이에 공유되는 메모리 양
    5. `buffers`: 커널이 시스템 성능 향상을 위해 사용하는 메모리 (주로 메타데이터 저장)
    6. `cached`: 페이지 캐시라고 불리며, I/O 속도 향상을 위해 사용되는 메모리
    7. `available` : `free + buffers/cached`를 단순 합산한 것보다 더 정교하게 "실제 할당 가능한 메모리"를 계산한 값
	    - `buffers`와 `cached` 영역에서도 당장 해제할 수 없는 부분들이 존재한다. 
		    - **잠긴 페이지 (Locked Pages):** `mlock()` 같은 시스템 콜을 통해 메모리에 고정된 페이지는 아무리 메모리가 부족해도 커널이 강제로 해제할 수 없다.
			- **공유 메모리 (Shared Memory):** `tmpfs`나 `shm` 영역은 `cached` 필드에 포함되지만, 이는 파일 내용을 단순히 복사해둔 것이 아니라 실제 데이터가 저장된 공간이므로 함부로 비울 수 없다.
			- **vm.min_free_kbytes:** 커널 파라미터로 설정된 이 최소 임계치는 시스템이 원활하게 작동하기 위한 **최후의 보루**값 이다. (리눅스 커널은 시스템 전체가 멈추는 것을 방지하기 위해 항상 일정량의 여유 메모리를 남겨둔다.)
			- -> `available`은 전체 캐시 양에서 이러한 "해제 불가능한 항목" 들을 미리 빼고 계산한 값이다.
	    - `avaliable`은 `buff/cache` 지표 중에서 즉시 반환될 수 있는 영역의 합이다.
	    - `-/+ buffers/cache` 라는 지표가 Mem 라인 아래에 존재하였는데 `avaliable` 지표로 바뀌었다
        
- ***Swap:**
    - 디스크의 일부를 메모리처럼 사용하는 공간으로, 메모리 부족 시 사용된다.

### 2. Buffers와 Cached 영역 

커널은 느린 디스크 I/O를 보완하기 위해 메모리의 일부를 캐싱 영역으로 할당한다.

![](images/Pasted%20image%2020260223234853.png)

- **Page Cache와 Buffer Cache의 차이:**
    
    - **Page Cache (Cached):** 
	    - 파일의 **내용(Contents)** 을 저장하는 캐시. 일반적인 파일 읽기/쓰기 시 사용된다.
	    - 커널이 파일 내용을 읽을 때, **bio 구조체**를 생성하여 디바이스 드라이버와 통신한다.
	    - 이 bio 구조체를 통해 디스크로부터 파일 내용을 읽어와 Page Cache 영역을 채우게 된다.
        
    - **Buffer Cache (Buffers):** 
	    - 파일 시스템의 **메타데이터(Super block, Inode block 등)** 를 저장하는 캐시
		    - super block: 
			    - 파일 시스템의 **전체적인 요약 정보**를 담고 있는 핵심 블록
			    - 파일 시스템의 크기, 빈 공간의 위치, 마운트 정보, 파일 시스템의 종류 등 해당 파티션의 "지도"와 같은 역할
			    - 슈퍼 블록이 깨지면 리눅스는 해당 파티션 전체를 인식하지 못하며 마운트 자체가 불가능해진다.
			- inode block:
				- 개별 **파일이나 디렉터리의 상세 정보**를 담고 있는 블록
				- 파일의 이름은 포함하지 않지만, 파일의 소유권(UID/GID), 접근 권한(Permissions), 해당 파일의 크기, 그리고 실제 데이터가 디스크의 어느 블록에 위치해 있는지에 대한 포인터 정보를 갖고 있다.
	    - 파일의 내용이 아닌 메타데이터를 읽을 때는 bio 구조체를 사용하지 않고, 대신 커널 내부 함수인 **`_get_blk()`** 와 같은 함수를 통해 블록 디바이스와 직접 통신하여 해당 내용을 Buffer Cache에 저장한다.
        
    - 두 영역 모두 I/O 성능 향상을 위해 존재하며, 메모리가 부족해지면 커널이 이 영역을 해제하여 프로세스에 반환한다.
        
- **시간에 따른 메모리 변화:**
    
    1. 초기에는 가용 영역(free)이 많음.
        
    2. 시간이 흐를수록 커널은 가용 영역을 캐시(Cache) 영역으로 전환하여 성능을 최적화함.
        
    3. 프로세스가 메모리를 더 요구하면, 커널은 캐시 영역을 줄이고 프로세스 사용 영역을 늘림.
        
    4. 캐시까지 다 쓰고도 부족하면 그때서야 **Swap**이 발생한다.

- **주요 항목 설명:**
    
    - **SwapCached:** 스왑으로 나갔던 메모리가 다시 물리 메모리로 돌아왔으나, 아직 스왑 영역에도 데이터가 남아있는 경우
        
    - **Active vs Inactive:**
        
        - **Active:** 최근에 참조되어 스왑 영역으로 이동되지 않을 메모리
            
        - **Inactive:** 참조된 지 오래되어 메모리 부족 시 스왑 영역으로 쫓겨날(Swap-out) 가능성이 높은 메모리
            
    - **Anon vs File:**
        
        - **Anon (Anonymous):** 
	        - 파일과 연결되지 않은 메모리. 즉, Page Cache 영역을 제외한 메모리 영역을 의미
	        - 주로 프로세스들이 사용하는 메모리 영역을 지칭할때 많이 사용된다. (프로세스의 힙, 스택 등)
	        - free 명령어의 used 에 해당됨.
            
        - **File:** 
	        - 파일과 연결된 메모리 (페이지 캐시 등). 
	        - 커널이 I/O 성능 향상을 위해 사용하는 영역으로, free 명령어에서 살펴본 buffers와 cached 영역이 여기에 속한다. 
	        - 부족하면 디스크로 다시 쓰거나 단순히 삭제(Drop)된다.
	          
	- Dirty:
		- `Dirty` 메모리는 메모리상에서 수정되었지만 아직 디스크에 기록되지 않은 데이터이다.
		- 양이 너무 많으면 시스템 크래시 시 데이터 손실 위험이 커지므로 `pdflush` 등의 커널 스레드가 주기적으로 디스크에 기록(Sync)한다.
            
- **LRU(Least Recently Used) List :**
    
    - 커널은 `Active`와 `Inactive` 리스트를 LRU 방식으로 관리한다.
    - 가장 오래전 사용된 페이지를 `Inactive`로 보내고, 끝까지 사용되지 않으면 메모리에서 해제하거나 스왑으로 보낸다.


### 4. Active 및 Inactive 메모리의 이동 

메모리가 부족해지면 커널은 LRU(Least Recently Used) 리스트를 확인하여 메모리를 확보한다.

- **핵심 원리: 부족함이 트리거가 되는 관리 메커니즘**
	- 커널은 단순히 **시간이 흐른다고 해서** 메모리를 `Active`에서 `Inactive`로, 혹은 `Swap` 영역으로 이동시키지 않는다.
	- 메모리 부족 현상이 발생하여 **해제해야 할 메모리를 찾아야 하는 순간**이 와야 커널은 비로소 LRU 리스트를 살펴본다는 것이 핵심이다.

- **메모리 해제 과정**: 커널 내부의 `kswapd` 데몬이나 `try_to_free_pages()` 함수가 작동하여 `Active` 리스트의 페이지를 `Inactive`로 옮기거나, `Inactive` 페이지를 해제하여 다른 프로세스에 할당한다.
    
- **테스트를 통한 확인**:
    
    - `malloc()`과 `memset()`을 통해 메모리를 할당하면 해당 영역은 `Active(anon)` 리스트에 머문다.
        
    - 단순히 시간이 흐른다고 해서 `Active` 메모리가 `Inactive`로 이동하지는 않는 것을 확인함.
        
    - **커널 파라미터 조작**: `vm.min_free_kbytes` 값을 높여 시스템의 최소 여유 메모리 임계치를 강제로 올리면, 커널은 메모리가 부족하다고 판단하여 `kswapd`를 실행시킨다.
        
- **결과**: `kswapd`가 활성화되면 `Active` 영역의 오래된 페이지를 `Inactive`로 옮기고 메모리를 해제하기 시작하며, 해제된 메모리는 `Swap` 영역으로 빠지기도 한다.

### 5. Slab 메모리 영역 

커널이 내부적으로 사용하는 메모리 영역을 **Slab**이라고 한다.
(`proc/meminfo`에서 확인 가능)

- **Slab의 구성**:
    
    1. **SReclaimable**: 재사용(해제) 가능한 Slab 영역. 캐시 용도로 사용되며 메모리 부족 시 해제되어 프로세스에 할당될 수 있다.
        
    2. **SUnreclaim**: 재사용 불가능한 영역. 커널이 현재 사용 중이며 강제로 해제할 수 없다.

```sh
[ec2-user@ip-172-31-4-90 ~]$ sudo slabtop -o
 Active / Total Objects (% used)    : 377370 / 380623 (99.1%)
 Active / Total Slabs (% used)      : 6803 / 6803 (100.0%)
 Active / Total Caches (% used)     : 114 / 171 (66.7%)
 Active / Total Size (% used)       : 50332.89K / 51520.69K (97.7%)
 Minimum / Average / Maximum Object : 0.01K / 0.13K / 10.19K

  OBJS ACTIVE  USE OBJ SIZE  SLABS OBJ/SLAB CACHE SIZE NAME                   
108630 108630 100%    0.02K    639      170      2556K avtab_node             
 45632  45632 100%    0.06K    713       64      2852K vmap_area              
 30296  30134  99%    0.07K    541       56      2164K Acpi-Operand           
 27783  27683  99%    0.19K   1323       21      5292K dentry                 
 27392  27355  99%    0.02K    107      256       428K kmalloc-16             
 20480  20436  99%    0.03K    160      128       640K kmalloc-32             
 20275  20275 100%    0.62K    811       25     12976K inode_cache         
```

- **Slab 할당자(Slab Allocator)의 필요성**:
    
    - 커널의 기본 메모리 할당 단위: **4KB(페이지)**
        
    - 하지만 커널 내부에서 사용하는 `inode cache`, `dentry cache` 등은 4KB보다 훨씬 작은 크기인 경우가 많다.
        
    - 4KB 단위로만 할당하면 메모리 파편화와 낭비가 발생하므로, Slab 할당자가 페이지를 목적에 맞는 작은 크기로 나누어 효율적으로 관리한다.
        
- **확인 방법**: `slabtop` 명령어를 통해 어떤 커널 캐시(예: `dentry`, `inode_cache` 등)가 메모리를 많이 사용하는지 실시간으로 확인할 수 있다.
	- ```sh
	[ec2-user@ip-172-31-4-90 ~]$ sudo slabtop -o | grep dentry
	 27762  27702  99%    0.19K   1322       21      5288K dentry  
	```

### 6. Case Study: Slab 메모리 누수

실제 운영 환경에서 발생할 수 있는 메모리 누수 사례를 다룹니다.

- **현상**: `free` 명령상 사용량(used)은 매우 높지만(예: 20GB), `ps` 명령으로 확인한 프로세스들의 메모리 합은 훨씬 적은(예: 10GB 미만) 상황이 발생했습니다.
    
- **원인 파악**: `/proc/meminfo` 확인 결과, `Slab` 영역 중 특히 `dentry cache`가 26GB를 점유하고 있었습니다.
    
    - `dentry cache`: 파일 시스템의 디렉터리 구조 및 부모/자식 관계를 저장하는 캐시입니다.
        
- **해결 및 분석**:
    
    - `drop_caches`를 통해 강제로 캐시를 플러싱(`echo 2 > /proc/sys/vm/drop_caches`)해도 잠시 후 다시 메모리가 차올랐습니다.
        
    - 분석 결과, `cron`에 등록된 `curl` 작업이 `nss-softokn` 라이브러리의 버그(잘못된 파일 참조)를 일으켜 `dentry cache`를 지속적으로 증가시킨 것이 원인이었습니다.
        
- **교훈**: 프로세스 사용량과 전체 사용량이 맞지 않을 때는 반드시 `/proc/meminfo`를 통해 `Slab` 영역의 누수를 의심해봐야 합니다.

### 요약

1. **buffers**: 파일 시스템의 메타데이터 등을 저장하는 블록 디바이스용 캐시입니다.
    
2. **cached**: I/O 효율을 위해 파일 내용을 저장하는 페이지 캐시 영역입니다.
    
3. **가용성**: 위 두 영역은 커널이 사용하지만, 프로세스가 필요로 할 때 언제든 해제하여 양보할 수 있는 영역입니다.
    
4. **anon & file**: `anon`은 프로세스 사용 메모리, `file`은 I/O 성능 향상을 위한 캐시입니다.
    
5. **LRU 관리**: `Active`와 `Inactive` 리스트를 통해 얼마나 최근에 참조되었는지를 기준으로 관리됩니다.
    
6. **Slab**: 커널이 직접 사용하는 캐시 영역으로 `dentry`, `inode`, `buffer_head` 등이 포함됩니다.
