## 1. 시스템 구성 정보 확인하기

### 1.1. 커널 및 부팅 메시지 확인 (`uname`, `dmesg`)

- `uname -a` 명령은 시스템의 커널 버전, 호스트네임, 시스템 아키텍처(x86_64, aarch64) 등 OS의 핵심 식별 정보를 출력한다.
    
- `dmesg` 명령은 시스템 부팅 시 커널이 하드웨어를 인식하는 과정에서 남긴 링 버퍼(Ring Buffer) 로그를 출력한다. AWS 환경에서는 인스턴스에 연결된 EBS 볼륨의 부착 상태를 확인하거나, 메모리 부족으로 인한 OOM(Out Of Memory) Killer 발생 기록을 사후 추적할 때 유용하게 쓰인다.
  
### 1.2. 하드웨어 물리적 정보와 가상화 환경 (`dmidecode`)

- **커널 이론:** `dmidecode` 명령은 서버의 DMI(SMBIOS) 테이블을 읽어 CPU 제조사, 메인보드 벤더, 메모리 뱅크 등의 물리적 하드웨어 정보를 출력한다.
    
- **AWS 환경 적용:** EC2 인스턴스에서 실행 시 물리적 장비 대신 Amazon EC2, Xen HVM, AWS Nitro System과 같은 하이퍼바이저 정보를 반환한다. 이를 통해 현재 인스턴스가 구형 Xen 기반인지 최신 칩셋(Nitro) 기반인지 파악하여 네트워크 및 EBS 스토리지의 성능 한계를 추정하는 기초 자료로 활용한다.
    

### 1.3. CPU 논리/물리 정보와 인스턴스 사이징 (`lscpu`, `/proc/cpuinfo)

- **커널 이론:** CPU 아키텍처, 소켓(Socket) 수, 코어(Core) 수, 스레드(Thread) 수 등 프로세서의 논리적, 물리적 구성과 L1~L3 캐시 크기를 확인한다.
    
- **AWS 환경 적용:** EC2의 '1 vCPU'는 일반적으로 물리적 코어의 '1개 하이퍼스레드'와 매핑된다. x86(Intel/AMD)인지 ARM(Graviton)인지 확인하고, 워크로드의 특성(캐시 의존도 등)에 맞춰 인스턴스 패밀리(C, M, R 등)를 선택하거나 하이퍼스레딩 비활성화 옵션 적용 여부를 결정하는 데 사용된다.
    

## 2. top을 통해 살펴보는 프로세스 정보들

### 2.1. 프로세스 메모리 지표 (VIRT, RES, SHR)와 컨테이너 OOM

- **커널 이론:** 
	* `VIRT`: 프로세스가 커널에 요청하여 할당받은 가상 메모리의 총량이다.
    
    - `RES`: 현재 물리 메모리에 실제로 올라와 사용 중인 실질적인 메모리 점유량이다.
        
    - `SHR`: 공유 라이브러리(glibc 등)나 공유 메모리 영역 크기이다.
        
- **AWS 환경 적용:** EKS나 ECS 환경에서 파드(Pod)/컨테이너의 메모리 제한(Limit)을 설정하고 모니터링할 때 기준이 되는 것은 `VIRT`가 아니라 `RES`이다. `VIRT`가 높더라도 `RES`가 Limit 이하라면 OOM(Out Of Memory) Killed가 발생하지 않으므로, 이 둘을 구분하여 불필요한 인스턴스 스케일업(Scale-up)을 방지해야 한다.
    

### 2.2. 프로세스 상태 (State)와 스토리지 병목

- **커널 이론:**
    
    - `R`: (running) 실행 중이거나 실행 큐에서 대기 중인 상태이다.
        
    - `S`: (sleep) 특정 이벤트를 기다리며 휴식 중인 상태이다.
        
    - `D (Uninterruptible Sleep)`: 디스크 I/O 등 중단할 수 없는 커널 작업의 완료를 기다리는 상태이다.
        
- **AWS 환경 적용:** 클라우드 스토리지는 네트워크 기반(EBS, EFS)이므로 로컬 디스크보다 지연(Latency)에 민감하다. `top`에서 `D` 상태 프로세스가 급증한다면, 이는 EC2 컴퓨팅 파워 부족이 아니라 연결된 EBS 볼륨의 IOPS/대역폭(Throughput) 제한에 도달했음을 의미한다.
    
### 2.3. 메모리 커밋(Memory Commit), 오버커밋(Overcommit)과 CoW

- **Memory Commit:** 프로세스가 `malloc()` 등을 통해 커널에게 메모리를 요청하면, 커널은 실제 물리 메모리(RES)를 즉시 주지 않고 가상 메모리(VIRT) 주소 공간만 약속(Commit)하는 행위이다.
    
- **Overcommit:** 시스템이 가진 실제 물리 메모리(Swap 포함)의 총량보다 더 많은 메모리를 프로세스들에게 커밋(약속)할 수 있도록 허용하는 리눅스 커널의 기본 정책이다. 프로세스들이 메모리를 요청만 해두고 당장 100% 사용하지 않는다는 통계적 특성을 이용해 리소스 효율을 극대화한다.
    
- **CoW (Copy-On-Write):** 프로세스가 `fork()`를 통해 복제될 때, 부모와 자식 프로세스는 동일한 메모리 페이지를 원본 그대로 공유한다. 그러다 둘 중 하나가 해당 페이지에 쓰기(Write) 작업을 시도하는 순간에만 페이지를 복사(Copy)하여 물리 메모리를 새로 할당한다. 이를 통해 메모리 낭비를 막고 프로세스 생성 속도를 비약적으로 높인다.

### 2.4. 프로세스의 우선순위: PR과 NI

- `top` 명령어를 통해 프로세스가 CPU를 할당받는 스케줄링 우선순위를 나타내는 지표인 PR과 NI 값을 확인할 수 있다.
    
- **NI (Nice):** 사용자(User) 레벨에서 설정할 수 있는 우선순위 값으로, -20부터 19까지의 범위를 갖는다. 값이 낮을수록 CPU를 먼저 할당받는 높은 우선순위를 의미하며, 기본값은 0이다.
    
- **PR (Priority):** 커널(Kernel) 레벨에서 관리하는 실제 스케줄링 우선순위이다. 일반 프로세스의 경우 `PR = 20 + NI`의 공식으로 계산되며(0~39 범위), `RT`로 표기되는 경우는 실시간(Real Time) 프로세스를 의미한다.
    
- 인스턴스의 CPU 리소스가 제한적이거나 경합이 심할 때, 중요한 백그라운드 데몬이나 모니터링 에이전트의 NI 값을 음수(예: -10)로 낮추어(우선순위를 높여) CPU 기아 상태(Starvation)를 방지하는 튜닝 기법으로 활용된다.

## 3. Load Average와 시스템 부하

### 3.1. Load Average의 정의와 Auto Scaling의 함정

- **Tick:** 리눅스 커널의 내장 타이머 인터럽트가 발생하는 주기 단위이다.
	- 커널은 매 Tick마다 **실행 큐 대기 프로세스(R)와 I/O 대기 프로세스(D)의 합**을 샘플링하여 1분, 5분, 15분 단위의 Load Average를 산출한다.
	- Load Average는 매 Tick(커널 타이머 인터럽트)마다 샘플링된 **실행 큐 대기 프로세스(R)와 I/O 대기 프로세스(D)의 합**의 평균값이다.
    
- **AWS 환경 적용:** 단순히 Load Average 수치나 CPU 사용률(%)만 보고 ASG(Auto Scaling Group)의 스케일아웃 정책을 동작시키는 것은 위험하다. I/O 대기(D 상태)로 인해 Load Average가 폭등한 경우, EC2 대수를 늘려도 백엔드 디스크 병목은 해결되지 않아 비용만 낭비된다.
    

### 3.2. 부하 원인 분석 (OS 지표와 CloudWatch 교차 검증)

- **커널 이론:** `vmstat 1`의 `r` (Run queue) 칼럼이 높으면 CPU 연산 부족, `b` (Block queue) 칼럼이 높으면 I/O 지연이다.
    
- **AWS 환경 적용:** `r` 큐가 높으면 EC2 vCPU를 늘려야 하지만, `b` 큐가 높다면 즉시 AWS CloudWatch로 넘어가 해당 인스턴스의 `EBSReadOps/WriteOps` 및 `VolumeQueueLength` 지표를 대조하여 디스크 확장 여부를 판별해야 한다.
    

## 4. free 명령이 숨기고 있는 것들

### 4.1. `free` 명령어의 기본 이해

- `free` 명령어는 시스템의 메모리 상태를 파악하는 가장 기본적인 도구이다. 전체 메모리(`total`), 사용 중인 메모리(`used`), 전혀 사용되지 않은 순수 빈 공간(`free`), 캐시 용도(`buff/cache`), 그리고 애플리케이션이 스왑 없이 즉시 사용할 수 있는 여유 메모리(`available`)를 출력한다.
    
- 메모리 부족 여부를 판단할 때 절대 단순 `free` 값을 봐서는 안 되며, **반드시 `available` 값을 기준으로 판단해야 한다.**

### 4.2. `free` 명령어와 Buffers vs Cached의 차이

- `free` 명령어의 `buff/cache` 영역은 디스크 I/O 최적화를 위한 공간으로 두 가지로 나뉜다.
    
    - **Buffers:** 파일 시스템의 메타데이터(Superblock, Inode 등)나 파일 시스템을 거치지 않는 블록 디바이스의 Raw I/O 데이터를 캐싱하는 영역이다.
        
    - **Cached (Page Cache):** 한 번 읽은 파일의 실제 내용(Data)을 메모리에 저장해 두어, 다음 요청 시 디스크를 거치지 않고 빠르게 제공하는 영역이다.
        
- **Right-Sizing:** `free` 공간이 적더라도 `buff/cache`가 크고 `available`(즉시 가용 메모리) 지표가 충분하다면 정상이다. 이를 기준으로 인스턴스 메모리 크기를 최적화해야 한다.

- **AWS 환경 적용:** `free` 명령에서 'free' 공간이 적다고 무조건 상위 인스턴스로 업그레이드해서는 안 된다. 커널이 캐시를 효율적으로 쓰고 있는 상태일 수 있으므로, 실제 응용 프로그램이 가용할 수 있는 'available' 지표를 기준으로 인스턴스 크기를 알맞게 유지(Right-Sizing)해야 비용을 최적화할 수 있다.
    
### 4.3. 메모리의 분류 (Anon vs File)와 LRU 리스트 관리

`/proc/meminfo`를 통해 시스템의 세부 메모리 상태를 파악할 수 있다.

- **Anon (Anonymous) vs File:**
    
    - **Anon Pages:** 프로세스가 동적으로 할당받은 힙(Heap)이나 스택(Stack)처럼, 파일 시스템의 특정 파일과 연결(Backing)되지 않은 순수 데이터이다. 돌아갈 원본 파일(고향)이 없으므로, 메모리 부족 시 이를 해제(방출)하려면 반드시 임시 대피소인 스왑(Swap) 영역(디스크)에 기록(Swap Out)해야만 한다. 시스템에 스왑 파티션이 없다면 커널은 이 페이지를 메모리에서 절대 쫓아낼 수 없다.
        
    - **File Pages:** Page Cache처럼 디스크의 특정 파일과 매핑된 메모리 영역이다. 데이터를 읽기만 한 상태(Clean Page)라면 디스크에 원본이 있으니 언제든 그냥 지워버려도(Drop) 무방하다. 데이터가 수정된 상태(Dirty Page)라도 스왑 영역으로 가는 것이 아니라, **원래 매핑되어 있던 원본 디스크 파일**에 변경점만 덮어쓰기(Write-back) 한 뒤 메모리를 비운다. 즉, File Page는 스왑 영역을 사용하지 않는다.
        
- **LRU (Least Recently Used) List와 Active vs Inactive:**
    
    - 커널은 메모리가 꽉 찰 경우를 대비해 페이지들을 LRU 알고리즘(이중 연결 리스트)으로 관리한다. 최근에 접근한 페이지는 `Active` 리스트에, 오랫동안 접근하지 않은 페이지는 `Inactive` 리스트에 둔다.
        
    - 메모리가 부족해지면(kswapd 가동), 커널은 `Inactive` 리스트의 끝에서부터 메모리를 방출(Eviction)하여 공간을 확보한다. 이때 `Inactive File`은 그냥 버리면 되지만, `Inactive Anon`은 디스크(EBS)의 Swap 공간으로 내보내야 하므로 심각한 I/O 부하를 유발한다.
        
### 4.4. Dirty Page와 동기화(Sync)

- **Dirty Page:** 프로세스가 메모리(Page Cache)의 데이터를 수정했지만, 아직 물리적 디스크에 기록(Write-back)되지 않아 디스크 원본과 내용이 달라진 상태의 페이지이다.
    
- 주기적으로 `pdflush` (최신 커널은 `flusher` 스레드)가 이를 디스크와 동기화한다. EBS 볼륨의 IOPS/대역폭 한계 상황에서 대량의 Dirty Page 동기화가 발생하면, I/O 처리가 밀리며 시스템 전체가 멈추는(Hang) 현상의 주원인이 된다.

### 4.5. Slab Allocator와 에이전트 누수

- 커널 내부 객체(dentry, inode 등)의 잦은 생성/소멸로 인한 파편화를 막기 위한 커널 전용 캐시 메모리(Slab)이다.
    
- 재사용 가능한 `SReclaimable`과 해제 불가능한 `SUnreclaim`으로 나뉜다. 인스턴스 내 모니터링 데몬이나 모듈에 버그가 있으면 `SUnreclaim`이 기하급수적으로 증가하며 메모리 누수(Leak)를 유발한다.
    
- **AWS 환경:** EC2에는 CloudWatch Agent, Datadog 등 다양한 모니터링 데몬이 백그라운드에서 동작한다. `SUnreclaim` 영역이 비정상적으로 비대해진다면 특정 커널 모듈이나 에이전트의 메모리 누수(Leak)를 의심해야 한다.
    

## 5. swap 메모리 증설의 포인트

### 5.1. 버디 시스템(Buddy System)과 메모리 재할당 과정

- **Buddy System:** 커널은 물리 메모리를 관리할 때 외부 단편화(Fragmentation)를 최소화하기 위해 메모리를 2의 거듭제곱 크기(페이지 단위)로 쪼개어 할당하는 버디 시스템 알고리즘을 사용한다.
    
- **메모리 재할당 과정:** 프로세스가 사용을 마치고 메모리를 반환하면, 커널은 인접한 빈 메모리 블록(짝, Buddy)을 찾아 다시 더 큰 연속된 메모리 블록으로 병합(Compaction)하는 재할당 과정을 거친다. 장기 가동되는 EC2 인스턴스에서 연속된 큰 메모리(Huge Page 등)가 필요할 때 이 병합 작업으로 인해 일시적인 CPU 스파이크가 발생할 수 있다.

### 5.2. 메모리 증설의 포인트 (Right-Sizing)

- 인스턴스에 메모리를 증설(Scale-up)해야 하는 정확한 시점은 단순히 `free` 메모리가 부족할 때가 아니다. 리눅스는 디스크 캐싱을 위해 남는 메모리를 적극적으로 사용하므로 `free`가 적은 것은 자연스럽다.
    
- **진짜 증설 포인트:** 애플리케이션이 즉시 사용할 수 있는 `available` 메모리가 지속적으로 고갈되고, `kswapd` 데몬이 활성화되어 스왑 활동(Swap Out/In)이 빈번하며, 이로 인한 디스크 I/O 상승으로 `D` 상태 프로세스가 급증할 때가 인스턴스 메모리 타입을 올려야 할 시점이다. (AWS에서는 CloudWatch Agent를 통해서만 이 OS 메모리 지표들을 정확히 추적할 수 있다.)
    

### 5.3. Swap In/Out과 커널 파라미터 `vm.swappiness`

- 메모리 부족 시 커널은 LRU 큐의 `Inactive Anon` 페이지를 스왑 영역(디스크)으로 내린다(Swap Out).
    
- AWS의 공식 AMI는 네트워크 디스크(EBS) 성능 고갈을 막기 위해 기본적으로 스왑 파티션이 없다. 불가피하게 스왑을 설정할 경우, `vm.swappiness` 값을 1~10으로 극단적으로 낮추어 EBS로의 페이징을 최대한 억제하고 데이터가 물리 메모리에 상주하도록 강제해야 한다.
    

### 5.4. 프로세스 디버깅 도구: gdb

- **gdb (GNU Debugger):** 프로그램의 실행 과정을 추적하고 메모리, 변수의 상태, 스택 트레이스를 확인할 수 있는 리눅스의 강력한 표준 디버깅 도구이다.
    
- 클라우드(EC2, EKS) 운영 환경에서 애플리케이션이 메모리 침범(Segmentation Fault)이나 OOM 등으로 비정상 종료되었을 때, 시스템이 남긴 코어 덤프(Core Dump) 파일을 `gdb`로 분석하여 정확한 장애 원인과 코드 라인을 역추적하는 데 필수적으로 사용된다.


## 6. NUMA, 메모리 관리의 새로운 세계

### 6.1. UMA vs NUMA 아키텍처와 대형 인스턴스

- 현대 고성능 서버는 각 CPU 소켓이 자신만의 로컬 메모리를 갖는 NUMA(Non-Uniform Memory Access) 구조이다. 로컬 메모리 접근(Local Access)이 원격 노드 메모리 접근(Remote Access)보다 빠르다.
    
- `m5.24xlarge` 등 엔터프라이즈급 대형 인스턴스는 물리 서버의 NUMA 토폴로지를 OS에 그대로 노출시키므로, 대용량 워크로드 운영 시 NUMA 병목을 반드시 고려해야 한다.
    
- **AWS 환경 적용:** `m5.24xlarge`, `x1e.32xlarge`와 같은 엔터프라이즈급 인스턴스나 베어메탈(Metal) 인스턴스는 물리 서버의 NUMA 토폴로지를 OS에 노출시킨다. 작은 인스턴스에서는 무시해도 되나, 대형 인스턴스 운영 시에는 필수 고려 대상이다.
    
### 6.2. NUMA 아키텍처의 메모리 할당 정책과 워크로드

NUMA 환경에서 리눅스 커널은 워크로드 특성에 따라 다음 세 가지 메모리 할당 정책을 지원한다.

- **Local 정책 (기본값):** 프로세스가 실행 중인 CPU 노드의 로컬 메모리에 우선 할당한다. 웹 서버나 마이크로서비스처럼 단일 노드 메모리로 충분히 커버되는 워크로드에서 메모리 접근 속도(캐시 지역성)를 극대화할 때 유리하다.
    
- **Interleave 정책:** 메모리를 여러 NUMA 노드에 라운드 로빈(Round-Robin) 방식으로 균등하게 분산 할당한다.
    
- **Bind 정책:** 지정된 특정 NUMA 노드에서만 메모리를 할당받도록 강제한다. 프로세스 격리가 필요한 극단적인 고성능 컴퓨팅(HPC) 환경에서 주로 사용된다.
    

### 6.3. Bind 정책의 동작 방식과 치명적 함정

- **동작 원리:** `numactl --membind` 명령 등을 통해 특정 노드에 프로세스의 메모리를 묶어둔다(Bind).
    
- **치명적 함정:** Bind된 노드의 메모리가 가득 차면, **시스템 전체에 다른 빈 노드(메모리)가 아무리 많이 남아 있어도 절대 빌려 쓰지 않고 즉시 Swap Out을 발생시키거나 OOM(Out Of Memory) Killed를 유발한다.** 이 특성을 모른 채 무분별하게 Bind를 설정하면 대형 장애의 원인이 된다.

### 6.4. Interleave 정책과 DB 워크로드 최적화

- **NUMA Imbalance:** Local 정책 사용 시, 프로세스의 덩치가 커져 특정 노드의 로컬 메모리만 100% 고갈되면 다른 노드의 메모리가 텅텅 비어있어도 Swap Out이 발생하는 불균형 현상이다.
    
- 대규모 메모리를 통짜로 사용하는 **In-Memory DB(Redis, SAP HANA)나 RDBMS** 운영 시 이 불균형이 잦은 지연(Latency)을 유발한다. 따라서 실행 시 `numactl --interleave=all` 옵션을 주어 모든 노드에 균등하게 메모리를 분산 배치하는 것이 표준 최적화 아키텍처이다.
    
- **AWS 환경 적용:** EC2에 Redis, SAP HANA, 대용량 RDBMS를 직접 호스팅할 때 NUMA Imbalance로 인해 쿼리 지연(Latency Spike)이 발생할 수 있다. 프로세스 실행 시 `numactl --interleave=all` 옵션을 주어 메모리가 모든 NUMA 노드에 균등하게 할당되도록 강제하는 것이 아키텍처 최적화의 핵심이다.
    
### 6.5. `numad`를 이용한 자동 메모리 할당 관리

- 수동으로 `numactl`을 사용해 각 프로세스의 노드를 지정하고 관리하는 것은 대규모 환경에서 매우 복잡하고 실수가 발생하기 쉽다.
    
- **`numad` (User-level daemon):** 백그라운드에서 동작하며 시스템의 NUMA 토폴로지와 워크로드의 CPU/메모리 사용 패턴을 실시간으로 모니터링한다. 최적의 지역성(Locality)을 유지할 수 있도록 프로세스와 메모리 페이지를 적절한 NUMA 노드로 **자동 마이그레이션**해 주는 편리하고 강력한 도구이다.


### 🚀 심무 지식 (완결형 보충 설명): 가상화 환경의 I/O 병목 추적 (OS와 AWS 지표의 결합)

**1. Demand Paging과 Major Page Fault의 연쇄 작용** 프로세스가 메모리를 요청할 때 커널은 가상 주소(VIRT)만 주고 물리 메모리(RES) 할당을 지연시키는 Demand Paging 기법을 쓴다. 이때 이미 메모리/캐시에 있는 데이터를 참조하면 **Minor Page Fault**가 발생해 성능 저하가 없지만, 물리 메모리에 데이터가 없어 디스크에서 직접 읽어와야 하는 상황이 되면 **Major Page Fault**가 발생한다. 디스크 I/O가 수반되므로 프로세스는 응답이 올 때까지 **D 상태(Uninterruptible Sleep)** 로 진입한다.

**2. AWS 환경에서 발생하는 "Hang(먹통)" 현상의 진짜 이유** 온프레미스의 로컬 SSD와 달리, AWS EC2에 연결된 EBS(Elastic Block Store)는 **네트워크 기반 논리적 스토리지**이다. Major Fault나 대량의 디스크 쓰기 작업이 발생하여 EBS로 요청이 넘어갔을 때, 사용 중인 EBS 볼륨의 **기본 제공 IOPS(예: gp3 기본 3,000)나 대역폭(Throughput) 제한에 도달**하게 되면 AWS 하이퍼바이저 층에서 즉각적으로 I/O 요청을 쓰로틀링(Throttling)한다. 이 순간 리눅스 커널 내부에서는 블록 I/O 대기열(`vmstat`의 b 큐)이 급격히 쌓이고, D 상태 프로세스가 누적되며, 시스템 Load Average가 코어 수의 수십 배로 치솟아 SSH 접속조차 불가능해지는 현상이 발생한다.

**3. 완벽한 교차 검증 및 트러블슈팅 가이드** "CPU 부하가 높다"는 경보가 울릴 때 맹목적인 EC2 스케일업은 오답이다. OS와 클라우드 인프라 지표를 연결하여 분석해야 한다.

- **[1단계] OS 내부 지표 확인:** `vmstat 1`에서 `b` 큐가 0 이상으로 치솟고, `top`에서 프로세스들이 `D` 상태이며, `iostat -x 1`에서 `%util`이 100%에 근접하거나 `await`(I/O 대기 시간) 값이 폭등하는지 확인한다.
    
- **[2단계] CloudWatch 지표 매핑:** 위 증상이 확인되면 AWS 콘솔에서 해당 EC2의 EBS 모니터링을 확인한다. 커널의 `b` 큐는 CloudWatch의 `VolumeQueueLength` 지표와 직접적으로 매칭된다. 추가로 `BurstBalance`(gp2) 고갈 여부나 쓰루풋 한계 도달 여부를 확인한다.
    
- **[3단계] 근본적 해결:** 이 병목의 해결책은 컴퓨팅 리소스 증설이 아니라 스토리지 성능 스케일업이다. 볼륨 유형을 프로비저닝된 IOPS(io2)로 변경하거나, gp3 볼륨 수정 기능을 통해 다운타임 없이 실시간으로 IOPS와 대역폭 수치를 늘려주는 것만으로 커널의 I/O 큐 정체를 즉시 해소할 수 있다.P
