	- `-c`: count

- **문제 1: Redis 통신 소켓의 TIME_WAIT 발생**
    
    - 상당량의 `TIME_WAIT` 상태 소켓이 6379 포트(Redis 서버)를 향해 생성되어 있음을 확인한다.
        
    - 원인: API 호출마다 Redis 서버와 연결(TCP 3-way handshake)을 맺고, 요청 완료 후 연결을 끊기 때문이다. 먼저 연결을 끊는 쪽(애플리케이션)에 `TIME_WAIT` 소켓이 남는다.
	    - 7장 그림
	      ![](images/IMG_9081.jpg)
        
    - 해결: 연결을 매번 맺고 끊지 않도록 **커넥션 풀(Connection Pool)** 방식을 적용하도록 코드를 수정한다. 
	    - TIME_WAIT
		    - **발생 이유**: TCP 프로토콜에서 연결을 종료할 때(4-way Handshake), **먼저 연결 종료(FIN)를 요청한 측(Active Closer)** 의 소켓이 최종적으로 진입하는 상태가 `TIME_WAIT`이다. 
			    - 네트워크 상에서 지연된 패킷이 뒤늦게 도착하여 데이터 무결성을 해치는 것을 방지하고, 정상적인 연결 종료를 보장하기 위해 약 1~2분(OS마다 다름) 동안 소켓을 유지한다.
			- **문제점**: 짧은 시간 동안 엄청나게 많은 요청을 처리하는 웹 서버나 API 서버에서 커넥션을 매번 맺고 끊으면(Short-lived Connection), 수많은 소켓이 `TIME_WAIT` 상태로 쌓이게 된다. 
				- 시스템에서 사용할 수 있는 로컬 포트(Ephemeral Ports, 일반적으로 32768~60999)는 한정되어 있으므로, `TIME_WAIT` 소켓이 포트를 모두 고갈시키면 새로운 연결을 맺지 못하는 **포트 고갈(Port Exhaustion)** 장애가 발생한다.
		- ```
	      7장 내용
	      
	      ## 7.2 TIME_WAIT 소켓의 문제점: 소켓 자원의 점유

		먼저 연결을 종료하려 시도한 **Active Closer** 쪽에는 **TIME_WAIT** 상태의 소켓이 생성된다. 
		이는 연결 종료 후에도 패킷의 유실이나 뒤늦은 도착에 대비해 소켓을 일정 시간 유지하는 상태이다.
		
		- **포트 고갈 문제**: 
			- 리눅스 커널은 외부와 통신 시 `net.ipv4.ip_local_port_range`에 정의된 범위 내의 로컬 포트를 할당한다. 
			- TIME_WAIT 소켓이 이 범위를 모두 점유하면 더 이상 할당할 포트가 없어 애플리케이션 타임아웃이 발생한다.
		    
		- **성능 저하**: 잦은 TCP 연결/종료는 반복적인 3-way handshake를 유발하여 응답 속도를 저하시킨다. 이를 방지하기 위해 **Connection Pool**과 같은 연결 재사용 방식이 권장된다.
	      ```
	    - ```python
	      코드 12-10 테스트용 애플리케이션 소스 코드 수정

			from flask import Flask
			app = Flask(__name__)
			
			pool = redis.ConnectionPool(host='infra-redis.redis.iwilab.com', port=6379, db=0)
			
			@app.route("/test/<key>")
			def testApp(key):
			    r = redis.Redis(connection_pool=pool)
			    r.set(key, time.time())
			
			    return r.get(key)
	      ```
        
    - 결과: `TIME_WAIT` 소켓이 1800개에서 700여 개로 줄어들고, 6379 포트의 `TIME_WAIT`는 완전히 사라진다. 
	    - 커넥션 풀 덕분에 TCP Handshake 오버헤드가 감소하여 초당 트랜잭션이 **267.95 trans/sec**로 향상된다.

### 12.5 nginx를 통해 reverse proxy 설정하기

- **Reverse Proxy 도입 배경**
    
    - 실무에서는 보안 설정의 유리함과 가상 호스트(`virtual host`), `server_name`을 통한 유연한 라우팅을 위해 gunicorn 앞단에 **nginx**나 **apache** 같은 프론트 서버(웹 서버)를 둔다.
        
    - **nginx 설치 및 기본 설정**
	    - 소스 코드를 컴파일하여 설치하며, TPS 확인을 위한 `--with-http_stub_status_module`과 디버깅을 위한 `--with-debug` 옵션을 포함한다. 
	    - `nginx.conf` 파일을 수정하여 80번 포트로 들어온 요청을 `127.0.0.1:5000` (gunicorn)으로 넘기도록(proxy_pass) 설정한다. 
		    - ```nginx
		       user  nobody;
				worker_processes  auto;
				
				events {
					worker_connections  1024;
				}
				
				http {
					include       mime.types;
					default_type  application/octet-stream;
				
					sendfile        on;
					keepalive_timeout  30;
				
					upstream gunicorn {
						server 127.0.0.1:5000;
					}
				
					server {
						listen       80;
						server_name  localhost;
				
						location / {
							proxy_pass http://gunicorn;
						}
					}
				}
		      ```
        
- **다섯 번째 테스트: 포트 고갈(Port Exhaustion) 에러 발생**
    
    - nginx를 거치도록 한 후 테스트 시, 
	    - 트랜잭션은 1556.10 trans/sec로 소폭 하락
	    - Availability(가용성)가 99.35%로 떨어져 일부 요청이 실패함.
        
    - nginx의 `error_log`를 확인하면 `Cannot assign requested address` 에러가 발생한 것을 볼 수 있었음.
	    - ```sh
	      [crit] 28394#0: *197158 connect() to 127.0.0.1:5000 failed (99: Cannot assign requested address) while connecting to upstream, client: 10.10.10.12, server: localhost, request: "GET /test/1 HTTP/1.1", upstream: "http://127.0.0.1:5000/test/1", host: "server"
	      ```
        
    - 원인: nginx가 gunicorn으로 요청을 전달할 때 사용할 **로컬 포트를 모두 소진(할당받지 못함)** 했기 때문
	    - TIME_WAIT 상태의 소캣을 확인해본 결과,
	    - nginx와 gunicorn 사이의 통신에서 대량의 `TIME_WAIT` 소켓이 발생함을 확인.

        
- **TIME_WAIT 소켓 문제 해결 방안**
    
    1. **커널 파라미터 튜닝 (`tcp_tw_reuse`)**
        
        - `sysctl -w net.ipv4.tcp_tw_reuse=1` 명령으로 `TIME_WAIT` 소켓을 재사용하도록 설정.
            
        - 테스트 결과 Availability가 100%로 복구되고 초당 트랜잭션이 상승.
            
    2. **Upstream Keepalive 설정 (소프트웨어적 해결)**
        
        - `tcp_tw_reuse`를 켜도 `TIME_WAIT` 소켓 자체는 여전히 많이 남는 것을 확인함.
	        - 근본적인 해결을 위해 nginx와 gunicorn(upstream) 사이에도 `keepalive` 통신을 설정한다.
            
        - `nginx.conf`의 upstream 블록에 `keepalive 1024;`를 추가하고, location 블록에 `proxy_http_version 1.1;`과 `proxy_set_header Connection "";`를 추가함.
	        - ```nginx
	          ... (중략) ...
				upstream gunicorn {
				    server 127.0.0.1:5000;
				    keepalive 1024;
				}
				
				server {
				    listen       80;
				    server_name  localhost;
				
				    access_log  logs/access.log  main;
				
				    location / {
				        proxy_pass http://gunicorn;
				        proxy_http_version 1.1;
				        proxy_set_header Connection "";
				    }
				}
	          ```
            
        - 일곱 번째 테스트 결과, 트랜잭션이 2077.94 trans/sec로 오르며 gunicorn 단독 구동 때와 비슷한 최고 성능을 보여줌을 확인
            
- **소프트웨어 한계치 튜닝 (worker_connections)**
    
    - 동시 접속자 수를 800명으로 늘려 가혹 테스트를 진행하면 `siege aborted due to excessive socket failure` 에러가 발생하며 테스트가 중단된다.
	    - ```sh
	      [root@client bin]# ./siege -c 800 -b -t30s -q -H "Connection: Keep-Alive" http://server/test/1
			[error] socket: read error Connection reset by peer sock.c:536: Connection reset by peer
			[error] socket: read error Connection reset by peer sock.c:536: Connection reset by peer
			[error] socket: read error Connection reset by peer sock.c:536: Connection reset by peer
			[error] socket: read error Connection reset by peer sock.c:536: Connection reset by peer
			      done.
			siege aborted due to excessive socket failure; you
			can change the failure threshold in $HOME/.siegerc
			
			Transactions:                   2046 hits
			Availability:                  66.64 %
			Elapsed time:                   1.30 secs
	      ```
        
    - nginx 에러 로그를 보면 `1024 worker_connections are not enough` 에러가 발생함을 확인
	    - ```sh
	       [alert] 7431#0: 1024 worker_connections are not enough
			[alert] 7431#0: 1024 worker_connections are not enough
			[alert] 7431#0: 1024 worker_connections are not enough
	      ```
        
    - 시스템 CPU/메모리 리소스는 남지만 소프트웨어 설정(nginx.conf의 worker_connections 기본값 1024) 때문에 요청을 처리하지 못한 것
        
    - 해결을 위해 `worker_connections`를 10240으로 늘리고, 이벤트 처리 모듈을 `select` 방식에서 더 성능이 좋은 `epoll` 방식으로 변경하며, `multi_accept on;` 옵션을 추가함.
	    - ```nginx
	      events {
			    use epoll;
			    worker_connections  10240;
			    multi_accept on;
			}
	      ```
        
    - 아홉 번째 테스트 결과 동시 접속 800명에서도 에러 없이 완벽하게 처리함을 확인했다.
        

### 12.6 요약

이번 장을 통해 애플리케이션 성능을 최적화하는 과정을 다음 4가지 핵심 원칙으로 정리할 수 있다.

1. 애플리케이션이 CPU 리소스를 최대로 활용하도록 **다수의 워커(최소 CPU 코어 수 이상)** 를 띄워 서비스해야 한다.
    
2. 대량의 **TIME_WAIT 소켓**은 성능 저하의 원인이므로, 발생 구간을 찾아 커넥션을 유지(Keepalive)하거나 재사용하도록 수정해야 한다.
    
3. 다른 서비스(DB, 캐시 등)와 연동할 때는 **커넥션 풀(Connection Pool) 방식**이나 **Keepalive**를 사용하여 연결 오버헤드를 줄여야 한다.
    
4. 시스템 리소스가 충분함에도 장애가 발생한다면, **애플리케이션(소프트웨어) 자체의 설정 한계(워커 수, 커넥션 제한 등)** 를 확인하고 늘려주어야 한다.
