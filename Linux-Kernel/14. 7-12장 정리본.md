## 7. TIME_WAIT 소켓이 서비스에 미치는 영향

### 7.1 TIME_WAIT의 발생 원리

- TCP 연결 종료를 위한 4-Way Handshake 과정에서 먼저 연결 종료를 요청한(Active Close) 단말의 소켓이 도달하는 최종 상태이다.
    
- 네트워크상에서 지연된 패킷(정체 패킷)이 뒤늦게 도착하여 신규 연결의 데이터와 섞여 발생하는 데이터 오염을 방지한다.
    
- 마지막 ACK 패킷이 유실되었을 때 상대방이 재전송하는 FIN 패킷을 받아 정상적인 종료를 보장하는 안전장치 역할을 수행한다.
    
- 리눅스 커널 내부적으로 2 MSL(Maximum Segment Lifetime) 동안 상태가 유지되며, 일반적으로 약 60초로 고정되어 작동한다.
    

### 7.2 서비스에 미치는 영향

- 서버가 외부 API나 데이터베이스(DB)와 통신하는 클라이언트 역할을 수행하거나 Nginx 같은 프록시(Proxy) 서버로 동작할 때, 트래픽 급증 시 대량의 아웃바운드용 TIME_WAIT 소켓이 적체된다.
    
- 소켓이 적체되면 신규 연결을 맺기 위해 할당해야 하는 아웃바운드 로컬 포트가 고갈되는 로컬 포트 고갈(Local Port Exhaustion) 현상이 발생한다.
    
- 로컬 포트 범위를 모두 소진할 경우 애플리케이션 레벨에서는 `Cannot assign requested address` 에러를 발생시키며 통신 장애로 이어진다.
    

### 7.3 커널 파라미터 튜닝

- `net.ipv4.ip_local_port_range`: 신규 아웃바운드 연결 시 할당할 수 있는 단기 포트(Ephemeral Port)의 범위를 정의하며, 범위를 확장하여 포트 고갈 시점을 늦출 수 있다.
    
- `net.ipv4.tcp_tw_reuse`: TIME_WAIT 상태의 소켓을 신규 아웃바운드 연결 시 재사용할 수 있도록 허용한다. 단, 통신하는 두 종단 모두 `net.ipv4.tcp_timestamps = 1` 설정을 통해 TCP 타임스탬프 옵션이 활성화되어 있어야만 안전하게 동작한다.
    
- `net.ipv4.tcp_max_tw_buckets`: 시스템이 동시에 유지할 수 있는 TIME_WAIT 소켓의 최대 개수를 제한한다. 임계치를 초과하면 커널은 초과된 TIME_WAIT 소켓을 즉시 파괴하며, 시스템 로그(`dmesg`)에 `TCP: time wait bucket table overflow` 경고 메시지를 기록한다.
    

[🚀 실무 지식 (완결형 보충 설명)]

- **개념 및 발생 이유:** TIME_WAIT은 분산 아키텍처(MSA) 및 고부하 웹 서비스 환경에서 잦은 내부 통신으로 인해 대량 누적되는 특성을 가진다. 과거 시스템에서 사용되던 빠른 소켓 회수용 파라미터인 `net.ipv4.tcp_tw_recycle`은 공인 IP 하나를 공유하는 NAT 환경의 클라이언트 패킷을 타임스탬프 불일치로 오인하여 드롭(Drop)하는 치명적인 문제를 유발하여 커널 4.12 버전부터 완전히 제거되었다.
    
- **실무적 의미:** 따라서 현대 리눅스 인프라 환경에서 아웃바운드 포트 고갈을 방어하는 표준 트러블슈팅 절차는 `tcp_tw_reuse`를 활성화(1)하고, 애플리케이션 레벨에서 Connection Pool 또는 HTTP Keep-Alive 설정을 강제하여 소켓 생성 및 소멸 자체를 최소화하는 구조적 제어를 동반해야 한다.
    

## 8. TCP Keepalive를 이용한 세션 유지

### 8.1 TCP Keepalive의 개념

- TCP로 연결된 두 종단 간에 물리적인 선로 단절이나 상대 노드의 비정상 종료(Dead Peer)를 감지하기 위해 백그라운드에서 주기적인 확인 패킷(Probe)을 무부하 상태로 전송하는 메커니즘이다.
    
- 경로 상에 위치한 방화벽(Firewall), NAT, L4 스위치 등의 네트워크 장비가 장시간 데이터 흐름이 없는 세션을 Idle 세션으로 판단하고 세션 테이블에서 강제 삭제(Session Timeout)하는 현상을 방지한다.
    

### 8.2 커널 파라미터와 동작 방식

- `net.ipv4.tcp_keepalive_time`: 커넥션이 Idle 상태가 된 후, 첫 번째 Keepalive Probe 패킷을 보낼 때까지 대기하는 시간이다. (리눅스 기본값 7200초 = 2시간)
    
- `net.ipv4.tcp_keepalive_intvl`: 첫 Probe 패킷에 대해 상대방의 응답이 없을 경우, 다음 후속 Probe 패킷을 전송하는 간격 시간이다. (리눅스 기본값 75초)
    
- `net.ipv4.tcp_keepalive_probes`: 응답을 받지 못했을 때 최종적으로 연결이 끊어진 것으로 간주하여 세션을 폐기하기 전까지 재시도하는 최대 패킷 전송 횟수이다. (리눅스 기본값 9회)
    

### 8.3 HTTP Keep-Alive와의 차이점

- L4 전송 계층에서 커널 스택이 주도하는 TCP Keepalive와 달리, 웹 서버(Nginx, Apache)가 제어하는 HTTP Keep-Alive는 L7 애플리케이션 계층에서 동작한다.
    
- HTTP Keep-Alive는 동일한 연결을 통해 여러 HTTP 요청/응답을 연속 처리하여 핸드셰이크 오버헤드를 줄이는 것이 목적이며, TCP Keepalive는 유휴 연결의 유효성을 지속 검증하고 유지하는 데 목적을 둔다.
    

[🚀 실무 지식 (완결형 보충 설명)]

- **개념 및 발생 이유:** 네트워크 단절이 발생했음에도 두 종단이 이를 인지하지 못하면 서버는 무한정 클라이언트의 응답을 기다리는 좀비 커넥션(Zombie Connection) 상태에 빠지게 되며, 이는 내부 리소스 누수로 직결된다.
    
- **실무적 의미:** 클라우드 환경(AWS ALB, NLB 등) 인프라 장비의 Idle Timeout 기본 임계치는 보통 60초 내외로 설정되어 있다. 리눅스 기본값인 2시간을 그대로 방치하면 중간 장비에 의해 세션 테이블이 이미 지워졌음에도 서버 소켓은 계속 연결된 것으로 오인하여 애플리케이션 행(Hang) 현상이 발생한다. 대안으로 `tcp_keepalive_time`을 60초 이하로 하향 조정하여 클라우드 장비의 만료 시간보다 빠르게 Probe를 발생시켜 세션을 보존해야 하며, 개발 시 애플리케이션 소켓 옵션(`SO_KEEPALIVE`)이 코드 내에서 명시적으로 활성화되었는지 반드시 검증해야 한다.
    

## 9. TCP 재전송과 타임아웃

### 9.1 TCP 재전송 메커니즘

- TCP는 패킷 송신 후 일정 시간 내에 수신 측으로부터 ACK를 받지 못하면 패킷 유실로 판단하고 동일 데이터를 재전송(Retransmission)한다.
    
- 재전송을 대기하는 기준 시간인 RTO(Retransmission Timeout)는 고정값이 아니며, 대상 서버와의 패킷 왕복 시간인 RTT(Round Trip Time) 데이터를 기반으로 가변 계산된다.
    
- 리눅스 커널의 최소 RTO 설정값(`TCP_RTO_MIN`)은 200ms(0.2초)로 하드코딩되어 있으므로, 아무리 대역폭이 넓고 빠른 로컬 네트워크 환경일지라도 패킷이 1개 유실되는 순간 무조건 최소 0.2초의 지연 시간이 발생한다.
    
- 재전송 시도가 반복될 때마다 RTO 대기 시간을 이전 대기 시간의 2배씩 증가시키는 지수 백오프(Exponential Backoff) 알고리즘을 사용하여 네트워크 혼잡을 제어한다.
    

### 9.2 주요 커널 파라미터

- `net.ipv4.tcp_retries1`: 네트워크 레이어(IP 라우팅 등)의 이상을 의심하고 내부적인 라우팅 재검토를 수행하기 전까지 시도하는 TCP 재전송 횟수이다. (기본값 3)
    
- `net.ipv4.tcp_retries2`: TCP 커넥션을 완전 폐기하고 애플리케이션 영역에 네트워크 에러를 반환하기 전까지 백그라운드에서 시도하는 최종 최대 재전송 횟수이다. (기본값 15, 최종 실패 처리까지 약 13분에서 30분이 소요된다.)
    
- `net.ipv4.tcp_syn_retries`: 최초 연결 요청 시 수립하는 SYN 패킷에 대한 최대 재전송 시도 횟수이다. (기본값 6)
    

[🚀 실무 지식 (완결형 보충 설명)]

- **개념 및 발생 이유:** 원격지 서버가 다운되거나 네트워크 경로 상의 장비 문제로 패킷이 드롭될 때, 클라이언트의 TCP 스택은 기본 설정에 의해 수십 분 동안 재전송 패킷을 쏘아대며 소켓 자원을 차단 상태로 유지한다.
    
- **실무적 의미:** 고가용성을 추구하는 마이크로서비스 환경(MSA)에서 특정 백엔드 노드의 장애가 전체 시스템으로 전파되는 현상을 막으려면 빠른 실패(Fail-fast) 전략이 요구된다. **애플리케이션은 타임아웃(예: Java Connection Timeout 3초) 시 즉시 포기하지만, 커널은 이를 모른 채 최대 30분(기본 15회) 동안 백그라운드에서 재전송을 시도한다. 이처럼 주기가 맞지 않으면 애플리케이션 로직이 끝났어도 커널에 좀비 소켓이 남아 로컬 포트 리소스 회수가 극심하게 지연된다.** 실무에서는 이러한 연쇄적 행(Hang) 예방을 위해 `tcp_retries2` 파라미터 값을 3~5(약 10~15초) 내외로 대폭 축소한다. **이를 통해 커널도 애플리케이션과 비슷한 시점에 미련을 버리고 소켓을 빠르게 단절시키게 만들며**, 즉시 확보된 자원과 서킷 브레이커 등을 통해 정상적인 이중화 노드로 트래픽을 우회시키는 튜닝을 적용한다.
    

## 9. TCP 재전송과 타임아웃

### 9.1 TCP 재전송과 RTO: 통신 신뢰성의 핵심 메커니즘

- TCP는 정해진 시간 내에 ACK를 받지 못하면 패킷 유실로 간주하고 데이터를 다시 보내는 재전송(Retransmission)을 수행한다.
    
- 패킷을 보낸 후 ACK를 기다리는 제한 시간을 **RTO(Retransmission Timeout)** 라고 한다.
    
- **InitRTO:** 최초 연결 시작(3-Way Handshake) 시 RTT를 알 수 없으므로 사용하는 초기값이며, 리눅스 커널에서는 기본 **1초**로 구현되어 있다.
    
- **RTO 계산 공식:** 이후 통신부터는 RTT(왕복 시간)를 기반으로 동적 계산된다.
    $$RTO=SRTT+4\times RTTVAR$$
    
- RTO의 최솟값($RTO\_MIN$)은 **200ms**로 설정되어 있어 내부망에서 5ms가 계산되어도 실제로는 200ms로 동작한다. 반면 해외망처럼 500ms가 넘어가는 환경에서는 계산된 RTO 값에서부터 지수적 백오프(Exponential Backoff)가 시작되므로 타임아웃이 기하급수적으로 길어진다.
    
- 세션별 RTO는 `ss -i` 명령어를 통해 확인할 수 있다. (`rto:214`는 214ms를 의미)
    

### 9.2 재전송을 결정하는 5가지 주요 커널 파라미터

리눅스 커널은 특정 파라미터에 따라 재전송 횟수와 간격을 조절한다.

- **1) `net.ipv4.tcp_syn_retries`**
    
    - 클라이언트가 서버에 연결을 요청하는 SYN 패킷의 재시도 횟수이다. (기본값 5)
        
    - 1초에서 시작해 2배씩 증가하여 총 31초(1+2+4+8+16)를 대기한다. 이를 **3**으로 줄이면 약 7초 만에 연결을 포기하므로 애플리케이션의 빠른 예외 처리가 가능하다.
        
- **2) `net.ipv4.tcp_synack_retries`**
    
    - 서버가 클라이언트에게 응답하는 SYN+ACK 패킷의 재전송 횟수이다. (기본값 5)
        
    - **SYN Flood 보안 연관:** 클라이언트가 마지막 ACK를 주지 않으면 서버는 SYN_RECV 상태로 SYN Backlog 큐를 점유하며 약 31초를 대기한다. 비정상 요청이 많으면 리소스가 고갈되므로 이를 적절히 줄여 방어해야 한다.
        
- **3) `net.ipv4.tcp_orphan_retries`**
    
    - 프로세스 연결이 끊어져 커널 소유가 된 소켓(Orphan Socket, 주로 `FIN_WAIT1` 상태)의 재전송을 제어한다.
        
    - **8회의 법칙:** 파라미터를 0으로 설정하더라도, 커널 내부 로직(`tcp_orphan_retries()`)에 의해 소켓의 RTO가 120초 미만(`alive = 1`)이면 강제로 **8회**(약 100초 대기) 재전송을 시도한다.
        
    - `tcp_fin_timeout`의 리눅스 표준 60초와 밸런스를 맞추어 정상적인 `TIME_WAIT` 도달을 유도하기 위해 **7**로 설정하는 것이 권장된다.
        
- **4) `net.ipv4.tcp_retries1` & `tcp_retries2`**
    
    - 연결이 성립된(ESTABLISHED) 상태에서의 재전송을 제어한다.
        
    - **tcp_retries1 (Soft Threshold, 기본 3):** IP 레이어에 라우팅 경로 확인을 요청하는 부드러운 경고 단계이다. (연결 유지)
        
    - **tcp_retries2 (Hard Threshold, 기본 15):** 재전송을 완전히 중단하고 소켓을 강제 종료하여 애플리케이션에 `ETIMEDOUT` 에러를 반환하는 최종 차단 단계이다.
        
### 9.3 RTO_MIN 값 변경하기 (빠른 재전송 튜닝)

- 내부망 통신처럼 극도로 빠른 환경에서는 200ms의 하드코딩된 $RTO\_MIN$이 지연의 원인이 될 수 있다.
    
- 시스템 전체 변경은 불가능하지만, 라우팅 단위로 `ip route change default via <GW> dev <Device> rto_min 100ms` 명령을 통해 최솟값을 변경하여 빠른 복구를 유도할 수 있다.
    

### 9.4 애플리케이션 타임아웃 권장값 (Connection vs Read)

- **Connection Timeout (3s 이상):** 최초 SYN 패킷 발송 후 1초 뒤에 1차 재전송이 일어나고, 그로부터 다시 2초 뒤(총 3초 경과 시점)에 2차 재전송이 발생한다. 커널이 최소 2번의 재전송 기회를 온전히 시도할 수 있는 시간을 보장하기 위해 애플리케이션 타임아웃을 최소 3초 이상으로 설정해야 한다.
    
- **Read Timeout (300ms 이상):** RTT 기반으로 계산되더라도 최소 보장값이 200ms이므로, 1회 유실을 견디려면 최소 300ms 이상으로 설정하는 것이 안전하다.
    

[🚀 실무 지식 (완결형 보충 설명)]

- **지수적 백오프의 이유:** 재전송 간격이 2배씩 늘어나는 것은 네트워크 혼잡(Congestion) 제어를 위함이다. 즉시 재전송하면 부하가 가중되므로 회복할 시간을 벌어주는 것이다.
    
- **RTO_MIN 튜닝 주의사항:** `rto_min`을 10ms 단위로 너무 낮게 잡으면, 일시적 네트워크 지연(Jitter)조차 유실로 오판하여 재전송을 남발하게 되므로 내부망 평균 RTT 모니터링 후 신중히 결정해야 한다.
    
- **Orphan 소켓과 메모리 임계치:** `FIN_WAIT1` 상태의 커널 귀속 소켓이 너무 오래 남아있으면 `tcp_mem` 임계치를 초과하여 신규 연결마저 실패할 수 있으므로 `ss -s`로 주기적 관찰이 필요하다.
    
- **동상이몽과 Fail-fast 튜닝 (핵심):** 고가용성 MSA 환경에서 특정 노드 장애 시, 애플리케이션은 타임아웃(예: 3초) 시 즉시 포기하지만 커널은 이를 모른 채 최대 30분(tcp_retries2 기본 15회) 동안 백그라운드에서 재전송을 시도한다. **이처럼 주기가 맞지 않으면 애플리케이션 로직이 끝났어도 커널에 좀비 소켓이 남아 로컬 포트 리소스 회수가 극심하게 지연된다.** * 실무에서는 연쇄적 시스템 정지(Hang) 예방을 위해 `tcp_retries2` 파라미터 값을 3~5(약 10~15초) 내외로 대폭 축소한다. **이를 통해 커널도 애플리케이션과 비슷한 시점에 미련을 버리고 소켓을 빠르게 단절시키게 만들며**, 즉시 확보된 자원과 서킷 브레이커 등을 통해 정상적인 이중화 노드로 트래픽을 우회시키는 튜닝을 적용한다.

## 10. dirty page가 IO에 끼치는 영향

### 10.1 Page Cache와 Dirty Page

- 리눅스 커널은 저속 블록 디바이스(디스크)의 접근 오버헤드를 극복하고자 물리 메모리의 여유 공간을 페이지 캐시(Page Cache)로 변환하여 관리한다.
    
- 애플리케이션이 쓰기(Write) 연산을 수행하면 데이터를 디스크로 직접 기록하지 않고 메모리의 페이지 캐시에 먼저 기록한다. 이때 메모리 상에는 데이터가 반영되었으나 물리 디스크에는 동기화되지 않은 상태의 메모리 블록을 더티 페이지(Dirty Page)라고 명명한다.
    

### 10.2 상태 모니터링 명령어

- `/proc/meminfo` 가상 파일을 조회하여 현재 시스템의 더티 페이지 통계치를 정밀 검증할 수 있다.
    
- `Dirty`: 현재 메모리에 적체되어 디스크로 내려가기를 대기 중인 실제 더티 페이지 용량이다.
    
- `Writeback`: 현재 커널 스레드에 의해 디스크 플러시(Flush) 작업이 실시간 진행 중인 가공 데이터 용량이다.
    
- 명령어 예시: `cat /proc/meminfo | grep -i dirty`
    

### 10.3 Dirty Page 동기화 메커니즘

- **백그라운 동기화 (`vm.dirty_background_ratio` / `vm.dirty_background_bytes`):** 시스템 전체 가용 메모리 중 더티 페이지가 차지하는 비중(또는 절대 바이트 용량)이 설정값(기본 비율 10%)을 초과하면 커널의 백그라운드 플러시 스레드(`flush/bdi_writeback`)가 기동된다. 이 동작은 백그라운드에서 진행되므로 애플리케이션 프로세스의 쓰기 연산을 차단하지 않는다.
    
- **포그라운드 동기화 (`vm.dirty_ratio` / `vm.dirty_bytes`):** 디스크의 쓰기 처리 한계로 인해 더티 페이지의 유입 속도가 방출 속도보다 빨라 동기화 상한 임계값(기본 비율 20%)에 도달하면, 커널은 애플리케이션의 모든 Write 시스템 콜을 강제 차단(Blocking)한다. 제어권을 빼앗긴 애플리케이션 스레드가 직접 자기가 생성한 더티 페이지를 디스크로 동기화하도록 강제 제어한다.
    

[🚀 실무 지식 (완결형 보충 설명)]

- **개념 및 발생 이유:** 대용량 메모리(예: 256GB RAM)를 탑재한 엔터프라이즈 서버에서 기본 비율인 `vm.dirty_ratio = 20` 설정을 사용하면 최대 약 50GB에 달하는 더티 페이지가 메모리에 축적된다. 이 거대한 용량이 한 번에 디스크로 플러시되는 임계점에 도달하면 스토리지 버스가 포화되어 심각한 IO 스파이크(Spike) 및 시스템 정지 현상이 유발된다.
    
- **실무적 의미:** 실무 환경의 대량 쓰기 유발 아키텍처(Kafka, MySQL, PostgreSQL 등)에서는 비율 기반 파라미터 대신 바이트 단위의 절대 용량 설정 파라미터인 `vm.dirty_background_bytes`와 `vm.dirty_bytes`를 도입하여 운용한다. 백그라운드 플러시 용량을 100~500MB 수준으로 낮추고 포그라운드 제한을 1~2GB 수준으로 제어하여, 적은 양의 데이터를 물리 디스크로 유연하고 끊임없이 플러시하도록 유도함으로써 애플리케이션이 차단되어 멈추는 지연(Latency) 현상을 근본적으로 차단한다.
    

## 11. IO 스케줄러

### 11.1 IO 스케줄러의 역할

- CPU 및 메모리 연산 속도에 비해 극단적으로 느린 블록 디바이스의 물리적 병목을 해결하기 위한 커널 계층의 최적화 엔진이다.
    
- 다수의 애플리케이션 프로세스로부터 접수된 무작위 디스크 읽기/쓰기 요청을 커널 내 디바이스 큐(Queue)에 일시 수집한 후, 물리적으로 인접한 섹터의 요청들을 하나로 묶는 병합(Merge)과 디스크 헤드의 이동 경로를 최소화하도록 순서를 재배치하는 정렬(Sort)을 수행한다.
    

### 11.2 알고리즘 분류

- **CFQ (Completely Fair Queuing):** 시스템 내 모든 프로세스에게 개별 IO 큐를 부여하고 가중치 기반 동등한 타임 슬라이스를 분배하는 방식이다. 탐색 시간(Seek Time) 오버헤드가 극심한 전통적 HDD 환경에 최적화된 설계 구조이다.
    
- **Deadline:** 요청 큐에 만료 시간(Deadline) 임계치를 할당하여 특정 IO 요청이 무한정 대기하는 기아 현상(Starvation)을 원천 방지한다. 읽기(Read) 작업에 쓰기(Write)보다 짧은 만료 시간을 부여하여 전반적인 대화형 성능을 보장한다.
    
- **NOOP (No Operation) / None:** 커널 레벨의 어떠한 복잡한 재정렬 연산도 수행하지 않고, 요청이 들어온 순서대로 디스크 컨트롤러에 즉시 밀어 넣는 FIFO(First-In, First-Out) 구조이다.
    

### 11.3 런타임 스케줄러 확인 및 동적 변경

- 서버 재부팅 과정 없이 `/sys` 가상 파일 시스템 가공을 통해 운영 중인 특정 디스크의 IO 스케줄러를 실시간 교체할 수 있다.
    
- **확인:** `cat /sys/block/sda/queue/scheduler` (출력 내용 중 대괄호 `[ ]` 표기된 항목이 현재 디바이스에 바인딩된 활성 알고리즘이다. 예: `noop [deadline] cfq`)
    
- **변경:** `echo noop > /sys/block/sda/queue/scheduler` (운영 환경에 즉각 반영되어 테스트가 가능하다.)
    

[🚀 실무 지식 (완결형 보충 설명)]

- **개념 및 발생 이유:** 기계식 헤더 이동 방식의 HDD 시대에는 물리 디스크 헤더의 이동 최소화가 성능의 절대적 지표였기에 CFQ 같은 복잡한 정렬 연산이 유리했다.
    
- **실무적 의미:** 플래시 메모리 기반의 고속 스토리지(SSD, NVMe) 환경은 데이터 접근에 따른 물리적 탐색 시간 자체가 존재하지 않는다. 오히려 플래시 스토리지 환경에서 커널 스케줄러가 IO 요청을 재정렬하기 위해 CPU 연산 자원을 소모하는 행위 자체가 시스템의 새로운 병목 지점으로 작용한다. 따라서 현대의 가상화 기반 클라우드 인스턴스 및 고성능 SSD 서버 환경에서는 IO 스케줄러를 `noop` 또는 `none` (멀티 큐 아키텍처 커널에서는 `none` 혹은 `mq-deadline`) 설정으로 튜닝하여 디스크 컨트롤러 자체의 병렬 처리 하드웨어 성능을 100% 이끌어내는 것이 업계의 표준 튜닝 가이드라인이다.
    

## 12. 애플리케이션 성능 측정과 튜닝

### 12.1 애플리케이션 만들기 및 성능 테스트 시작

- 성능 최적화의 베이스라인 측정을 위해 Python Flask(REST API)와 Redis를 활용하여 타임스탬프를 입출력하는 구조를 만든다.
    
- `siege` 벤치마킹 툴을 이용해 테스트를 진행한다. (`siege -c 100 -b -t30s -q http://server/test/1`)
    
- 초기 테스트 결과, 플라스크의 내장 웹 서버는 싱글 스레드로 동작하여 **33.52 trans/sec**의 낮은 처리량을 보여주었다.
    

### 12.2 CPU 성능 최적화하기

- `top` 명령어로 확인 시 단 하나의 `python` 프로세스만 동작하며 단일 코어만 소모하고 있음을 확인한다.
    
- 내장 웹 서버 대신 멀티 프로세스를 지원하는 WSGI 서버인 **gunicorn**을 도입한다.
    
- `gunicorn -w 4` 옵션으로 워커 프로세스를 CPU 코어 수에 맞게 늘리자 초당 트랜잭션이 **140.77 trans/sec**로 비약적으로 상승한다.
    

### 12.3 네트워크 소켓 최적화하기

- `ss -s` 및 `netstat -napo` 명령어로 소켓을 분석한 결과 병목 지점을 두 곳 발견한다.
    
- **문제 1 (Redis 소켓 TIME_WAIT):** 매 API 호출마다 Redis와 TCP 연결을 맺고 끊어 발생한다.
    
    - **해결:** 애플리케이션 코드에 커넥션 풀(Connection Pool)을 적용하여 TCP Handshake 오버헤드를 제거, **267.95 trans/sec**로 향상된다.
        
- **문제 2 (웹 서버 클라이언트 TIME_WAIT):** 서버(gunicorn)가 응답 후 `Connection: close`를 내려보내며 연결을 단절시킨다.
    
    - **해결:** gunicorn의 워커를 HTTP/1.1 Keep-Alive를 지원하는 비동기(`async`) 워커인 **eventlet**으로 변경(`-k eventlet`)하고, `--keep-alive 10` 옵션을 준다.
        
    - 클라이언트 도구(`siege`)에도 `-H "Connection: Keep-Alive"` 헤더를 추가하여 양방향 연결 유지를 적용한 결과 **1917.20 trans/sec**로 극적인 성능 향상을 달성한다.
        

### 12.4 Nginx를 통한 리버스 프록시 튜닝

- 보안과 유연한 라우팅을 위해 gunicorn 앞단에 nginx를 리버스 프록시로 배치하고 테스트하자, `Cannot assign requested address` 에러와 함께 포트 고갈(Port Exhaustion)이 발생한다. (nginx와 gunicorn 사이의 잦은 통신으로 인한 TIME_WAIT 누적)
    
- **해결 1 (커널 튜닝):** `sysctl -w net.ipv4.tcp_tw_reuse=1`로 소켓 재사용을 허용하여 가용성을 100%로 복구한다.
    
- **해결 2 (Upstream Keepalive):** nginx 설정 파일에 `keepalive 1024;`, `proxy_http_version 1.1;`, `proxy_set_header Connection "";`를 추가하여 내부 구간에도 연결 유지를 적용, **2077.94 trans/sec**를 기록한다.
    
- **해결 3 (소프트웨어 한계 튜닝):** 동시 접속을 800으로 올리자 `1024 worker_connections are not enough` 에러가 발생한다. nginx의 `worker_connections`를 10240으로 늘리고 `use epoll; multi_accept on;`을 설정하여 가혹 조건도 완벽하게 방어해낸다.
